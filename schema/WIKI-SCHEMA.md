## 📖 概述

这是知识 Wiki 系统的「宪法文件」，定义了 Wiki 的组织规则、页面模板、流程规范。

所有 Wiki 系统 Agent 和 Notion AI 都应遵循此文件。各 Agent 的详细职责分工见 系统工作流程图。

---

## 📂 条目类型定义

### concept（核心概念词条）

页面结构：

- **定义**：一句话说明白这个概念是什么

- **关键要点**：3-5 个核心要点

- **实例/应用**：具体使用场景或案例

- **关联概念**：与其他 concept 页面的 `<mention-page>` 链接

- **来源引用**：引用的 summary 条目列表

### summary（单篇资料摘要）

页面结构：

- **一句话摘要**：文章核心观点

- **关键洞察**：3-5 个最重要的发现/观点

- **提取的概念**：使用 `<mention-page>` 标签引用对应的 concept 页面（不是纯文本）

- **原始文章信息**：作者、来源、发布时间、原文链接

- **个人评注**：与 Tizer 现有工作流的关联点

### synthesis（跨资料综合分析）

**何时创建**：

- 某个标签下积累了 **≥10 个 concept**，且没有已有 synthesis 覆盖该主题

- 或者两个标签的交叉领域积累了足够深度（如「Agent 编排 × 记忆系统」）

- 或者一轮系统性对话产出了重要的架构决策

**每次最多生成 2 篇**，深度优先。

页面结构：

- **研究问题**：这篇 synthesis 要回答什么问题

- **综合分析**：跨多篇资料的综合推理（鼓励使用对比表）

- **关键发现**：3-5 个跨来源才能发现的洞察

- **来源列表**：引用的所有 summary 和 concept

- **行动建议**：基于分析的可执行建议

### qa（问答沉淀）

**执行者**：Wiki QA Agent

**何时创建**：当回答综合了 **≥3 个 Wiki 页面**的内容，且问题不是一次性的操作请求（如「帮我合并这两个概念」）。

页面结构：

- **问题**：原始提问

- **回答**：综合回答正文

- **引用来源**：使用 `<mention-page>` 引用的 Wiki 页面列表

### entity（实体档案 — 人物/产品/公司的持续跟踪页）

**与 concept 的区别**：

- concept = 技术概念、方法论、架构模式（如 Harness 工程、ReAct Agent、RAG）

- entity = 具体的产品、公司、人物（如 OpenClaw、Anthropic、Karpathy）

**判断标准**：如果这个概念有版本历史、融资信息、GitHub Stars、团队背景等「档案属性」，它就是 entity 而不是 concept。

页面结构：

- **简介**：一句话介绍这个实体

- **关键数据**：官网、GitHub、Stars、License、团队、融资等

- **版本/动态时间线**：重要版本发布和事件记录（持续更新）

- **与我的关系**：Tizer 如何使用/关注这个实体

- **相关 concept**：与此实体相关的技术概念 `<mention-page>` 链接

- **所有引用的 summary**：所有提到这个实体的摘要列表

### lint-report（健康报告）

页面结构：

- 检查日期、健康度评分、孤岛条目、过期草稿、重复疑似、状态异常、标签异常、标签分布统计、状态晋升建议、改进建议

## 📨 对话产出沉淀规则

当 Notion AI 与用户的对话产出了有价值的结果时，按以下规则判断是否需要沉淀到 Wiki：

| 对话类型 | 是否入 Wiki | 沉淀方式 |

| --- | --- | --- |

| **知识检索**（“X 是什么？”） | ✅ 是（综合≥3页时） | 由 Wiki QA Agent 回答并自动创建 `qa` 页面 |

| **系统设计/架构决策** | ✅ 是 | 提炼决策备忘录为 `synthesis`，规则本身已物化到 Agent 指令 |

| **日常维护执行**（合并重复、晋升状态） | ❌ 否 | Lint Report 已记录，无需额外页面 |

| **头脑风暴/探讨** | 视情况 | 有结论的提炼为 concept 或 synthesis，纯发散不入库 |

## 🔄 Ingest 流程（资料编译）

**执行者**：Wiki Compiler Agent

**触发条件**：新文章入库（page.created，60s 防抖）或手动 @mention。~~定时批量触发已关闭（存量已清零）~~

**执行步骤**：

1. 读取新文章的完整内容和备注

1. **🔒 Summary 去重检查**：用源文章 URL 查询 Wiki 是否已存在对应 summary，命中则跳过（兑底用标题模糊匹配）

1. 创建一条 **summary** 页面（状态=已审核）

1. 从文章中提取核心概念（3-7 个）

1. 对每个概念：

  - **强制查重**：先将名称归一化（转小写、去空格、去标点），用 SQL 精确匹配 + 模糊匹配查询已有 concept

  - 已有 → 只追加引用，不创建新页面

  - 没有 → 创建新 concept 页面（状态=草稿）

1. 提取的概念用 `<mention-page>` 回填到 summary 的「提取的概念」段

1. 设置所有新页面的标签、置信度、最后编译时间

1. 标记源文章 `已编译到Wiki = true`

**重要原则**：

- 增量更新，不要全量重写已有页面

- concept 名称使用业界通用术语，遵循命名归一化规则

## 🔍 Query 流程（检索回答）

**主要执行者**：Wiki QA Agent（知识库专属问答窗口）、Notion AI（兑底 + 系统管理场景）

**其他 Agent 也使用 SQL 查询来完成各自职责**（Compiler 查重、Lint 统计、Fixer 定位目标等）。

**查询策略（按优先级）**：

1. **SQL 直查**：用 `querySql` 按名称、标签、类型等条件查询数据库，这是最快最准的方式

1. **读取具体页面**：用 `loadPage` 读取 concept/summary/synthesis 的完整内容

1. **全局搜索**：如果 SQL 查询结果不足，用 `search` 跨所有来源搜索

**回答流程**：

1. 根据问题类型选择查询策略，定位相关页面

1. 读取页面完整内容

1. 综合推理，标注来源引用。**根据来源条目状态差异化处理**：

  - **已审核**：优先引用，作为核心论据

  - **审核中**：正常引用

  - **草稿**：标注「此概念为新建条目，内容可能不完整」

  - **需更新**：标注「此内容可能已过时，建议交叉验证」

1. 如果回答具有沉淀价值，按「对话产出沉淀规则」判断是否创建 qa 或 synthesis

**常用 SQL 模板**：

- 按标签找 concept：`SELECT url, "名称" FROM wiki WHERE "类型" = 'concept' AND "标签" LIKE '%记忆系统%'`

- 按名称模糊查找：`SELECT url, "名称", "类型" FROM wiki WHERE "名称" LIKE '%MemBrain%'`

- 查看最近更新：`SELECT url, "名称", "类型" FROM wiki ORDER BY "最后编辑时间" DESC LIMIT 10`

- 查看已有 synthesis：`SELECT url, "名称" FROM wiki WHERE "类型" = 'synthesis'`

## 🩺 Lint 流程（健康检查）

**执行者**：Wiki Lint Agent

**触发条件**：recurrence 每日 14:00 自动执行，或手动 @mention

### Phase 1：快速检查（纯 SQL，全量覆盖）

不需要 loadPage，全部通过 `querySql` 完成，保证 100% 条目覆盖：

1. **全库统计** — 按类型×状态交叉统计

1. **过期草稿检测** — 草稿超过 **7 天**：内容完整 → 建议晋升为审核中；内容不完整 → 报警

1. **时效性检查** — 最后编译时间超过 **30 天**或为 null（豁免 qa 类型和系统元概念）

1. **完全同名重复检测** — `GROUP BY "名称", "类型" HAVING COUNT(*) > 1`

1. **近似重复检测（归一化）** — 分两步：

  - Step A：SQL 查出所有 summary/concept/entity 的名称和 url

  - Step B：在推理层做归一化匹配（lowercase → 去标点/空格/句号 → 统一繁简体 → 去版本号后缀），输出疑似重复对

1. **状态异常** — 状态为空的页面

1. **标签异常** — 标签为空或使用已废弃标签 + 全量标签分布统计

1. **类型启发式筛查** — SQL 全量查出 concept 名称，在推理层用规则批量筛查「疑似应为 entity」：

  - 规则 A：名称含版本号（`2.0`/`v3`）或产品后缀（`Tool`、`SDK`、`CLI`等）

  - 规则 B：首字母大写英文专有名词、大写缩写+产品名组合

  - 输出疑似列表，标注「建议人工确认」

### Phase 2：深度检查（需 loadPage，抽样覆盖）

从 Phase 1 发现的问题条目 + 随机抽样中执行：

1. **引用结构化检查** — 抽样 20 个草稿 concept/entity，检查来源引用是否使用 `<mention-page>` 结构化链接

1. **标签分类合理性抽查** — 加载 Phase 1 筛查标记的条目 + 随机抽样，人工判断标签匹配度

1. **状态晋升评估** — 综合 Phase 1 数据，判断哪些条目应晋升（草稿→审核中：7天+内容完整；草稿/审核中→已审核：≥2篇引用；审核中/已审核→需更新：>30天）

### 输出规范

lint-report 必须包含：

- 健康评分（0-100）+ 计分明细表格

- 各检查项的发现（含具体页面链接）

- **Fixer 可执行的结构化修复清单**（每条标注：修复类型 + 目标页面 URL + 具体操作）

- 待办建议（需人工介入的项目）

- 报告末尾 @mention Wiki Fixer 触发自动修复

### Fixer 协作协议

- 建议分两类：🤖 **自动修复项**（重复合并、标签/类型修正、引用结构化升级）+ 👤 **人工介入项**（Schema 变更、豁免规则、Compiler 调整）

- Fixer 完成后应在 Lint Report 追加「修复记录」区块，列出每项修复的操作和结果

- Recheck 时，Lint Agent 只需读取修复记录 + 验证变更页面属性，无需全量重扫

**SQL 规范**：不允许自连接、多条语句、动态 LIKE 拼接。必须分步查询（先查 concept/entity 列表，再查 summary 列表），在推理层做匹配。

## 📝 命名规范

- **concept**：英文产品名跟官方大小写一致；中文用最短准确名；不加括号说明；版本号不单独建页；别名选一个标准名做标题

- **summary**：「摘要：」+ 原文标题

- **synthesis**：研究问题作为标题

- **entity**：实体官方名称（产品用官方写法如 OpenClaw、LangGraph；人物用常用英文名如 Karpathy；公司用官方名如 Anthropic）

- **lint-report**：「Lint Report YYYY-MM-DD HH:mm」（必须含时分，避免同一天多份报告同名）

## 🏷️ 双层标签体系

### 一级标签（14 个，粗分类）

每个 **concept、entity 和 summary** 选 1-2 个最相关的一级标签：

- **Agent 框架**：具体 Agent 产品/平台/OS

- **Agent 编排**：驾驭 Agent 的方法论/架构模式/多 Agent 协调

- **Agent 技能**：Agent 使用的 Skill/Plugin/工具链

- **Coding Agent**：AI 编程 Agent 和编码辅助工具（Claude Code、Codex、Gemini CLI、CodeBrain）

- **LLM**：模型架构/训练技术/推理优化的理论概念（具体模型产品归 entity）

- **记忆系统**：Agent 记忆、上下文管理、知识图谱

- **OpenClaw**：OpenClaw 生态专有概念（技术概念用 concept，生态产品用 entity）

- **Crypto/DeFi**：链上 Agent、DeFi 协议、Web3

- **知识管理**：知识库、Wiki 方法论、信息管理

- **开发工具**：传统基础设施工具（网络/容器/版本控制/本地模型运行）——**不包含 AI 编程 Agent（那些归 Coding Agent）**

- **工作流**：仅非 Agent 的自动化流程、变现路径

- **安全/隐私**：Agent 安全、隐私计算、沙箱、信任机制

- **内容创作**：AI 生成视频、音乐、设计、动画

- **商业/生态**：Agent 经济模型、市场、支付、分发

**已废弃标签（禁止使用）**：~~AI Agent~~、~~MCP~~、~~Notion~~

### 二级标签（来源标签，细粒度）

存储在「来源标签」属性中，是从源文章数据库（微信文章/X书签）透传的原始标签，不做映射和归一化。

- **用途**：保留原始信息供查阅和追溯，支持细粒度 SQL 搜索

- **格式**：通字符串，如「OpenClaw, Agent, LLM, 自动化, Claude, 记忆」

- **维护者**：Compiler 在编译时自动从源文章的标签属性复制

## 🤖 Agent 职责速查

| Agent | 角色 | 核心职责 | 可修改范围 |

| --- | --- | --- | --- |

| **Wiki Compiler** | 编译器 | 文章 → summary + concept | 创建新 summary/concept，追加引用 |

| **Wiki Lint Agent** | 诊断师 | 每日 14:00 健康检查 → lint-report | 只创建报告，不修改任何页面 |

| **Wiki Fixer** | 治疗师 | 按 Lint Report 自动修复 7 类问题 + 引用链修复 + 指令模式 | 状态/标签/类型修复、同名重复合并、近似重复合并、引用链修复（孤岛 + 纯文本升级）；不得修改 index 类型系统页；删除需人类指示 |

| **Wiki Synthesizer** | 知识进化者 | 发现主题聚集 → synthesis | 只创建 synthesis，不修改其他页面 |

| **Wiki QA Agent** | 知识问答窗口 | 接收提问 → 查 Wiki → 综合回答 → 创建 qa | 读取所有 Wiki 内容；创建 qa 页面 |

| **Notion AI** | 协调者 + 系统管理员 | Schema 设计、Mem0 桥接、复杂协调、兑底 Query | 全部（兑底角色） |

各 Agent 的完整触发机制和详细职责见 系统工作流程图。

## 🔄 变更同步规则

当任何 Agent 的指令页面被修改时，必须同步检查并更新以下文件：

1. **本文件（Wiki Schema）** — 检查「Agent 职责速查」表格、流程规范是否需要更新

1. 系统工作流程图 — 检查 Agent 职责清单、架构图、自动化运行机制是否需要更新

1. **Mem0** — 存储变更摘要供跨会话追溯

**谁负责**：执行修改的角色（Notion AI 或用户）在修改 Agent 指令后，应立即执行同步。不允许「只改指令不同步宗法文件」。
