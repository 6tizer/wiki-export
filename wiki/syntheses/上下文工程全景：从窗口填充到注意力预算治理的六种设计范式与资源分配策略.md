## 研究问题

当 Agent 从单轮问答走向长时程自主执行，上下文窗口从「足够大」变成「永远不够用」。**上下文管理的本质是什么？它在不同架构层级呈现出哪些设计范式？各范式之间的张力与互补关系如何？**

## 综合分析

### 一、上下文管理的六层设计范式

上下文管理并非单一技术，而是一组按干预深度递增排列的设计范式，从最轻量的被动填充到最重量的主动重构：

| 层级 | 范式 | 核心机制 | 代表概念 | Token 成本 | 信息损失 |

| --- | --- | --- | --- | --- | --- |

| L0 | **被动填充** | 将系统提示、工具定义、文件内容一次性装入窗口 | [Untitled](concepts/OpenClaw 九层 System Prompt 架构.md) | 高（固定开销） | 无 |

| L1 | **前缀缓存复用** | 利用静态前缀的 KV Cache 避免重复计算 | [Untitled](concepts/Prompt Caching.md)、[Untitled](entities/SGLang.md) | 低（缓存命中时） | 无 |

| L2 | **渐进式截断** | 按时间或重要性丢弃旧消息，保留近期上下文 | [Untitled](concepts/Claude Code 上下文管理.md) 第 1-2 层 | 中 | 低 |

| L3 | **摘要压缩** | 用 LLM 将历史对话压缩为摘要，再注入上下文 | [Untitled](concepts/Claude Code 上下文管理.md) 第 3-5 层、[Untitled](concepts/suggest-compact hook.md) | 中（压缩成本 + 注入成本） | 中 |

| L4 | **外部化检索** | 将知识/记忆存储到外部系统，按需检索注入 | [Untitled](concepts/Context Agent.md)、[Untitled](concepts/Agentic Navigation.md)、[Untitled](concepts/LLM 知识编译.md) | 低（按需加载） | 取决于检索质量 |

| L5 | **架构分治** | 通过多 Agent/子任务拆分，让每个 Agent 只持有局部上下文 | [Untitled](concepts/thin control over thick state.md)、[Untitled](concepts/Recursive Language Model.md) | 总量更高，但单 Agent 可控 | 跨 Agent 信息丢失风险 |

### 二、两条核心张力轴

**张力一：信息完整性 vs. 计算效率**

L0-L1 追求零信息损失但成本高昂；L2-L3 通过有损压缩降低成本；L4-L5 试图在两者间找到最优解。这条轴上的设计选择，本质是在回答一个问题：**哪些信息值得占用注意力预算？**

[token效率](concepts/token效率.md) 揭示了一个关键洞察：同一任务下不同模型的 token 消耗差异巨大，这意味着上下文管理不仅是「窗口管理」，更是「注意力经济学」——每个 token 都有机会成本。

**张力二：确定性控制 vs. 自适应演化**

静态的 System Prompt 架构（如 OpenClaw 九层结构）提供可预测的上下文布局，但无法应对运行时的动态变化。而自适应机制（如 [Claude Code 上下文管理](concepts/Claude Code 上下文管理.md) 的 6 层渐进式降级）则根据窗口压力动态选择策略，但增加了系统复杂度和不可预测性。

[Scope Reduction Detection](concepts/Scope Reduction Detection.md) 代表了一种折中路线：在自适应框架中嵌入确定性护栏，监控上下文是否偏离预期范围。

### 三、上下文管理的三个演进阶段

1. **窗口填充期（2023-2024）**：核心问题是「如何把有用信息塞进去」，技术焦点在 RAG、向量检索、长上下文模型

1. **压缩治理期（2025-2026初）**：核心问题是「如何在窗口内保持信息质量」，技术焦点在摘要、截断、缓存策略

1. **注意力预算期（2026-）**：核心问题是「如何让每个 token 产生最大价值」，技术焦点从窗口管理转向注意力资源的经济学优化

[OpenClaw Context Engine](entities/OpenClaw Context Engine.md) 是这一演进的缩影：从最初的对话工具（窗口填充）→ 流式场景的上下文维护（压缩治理）→ 持续执行 Agent 的上下文基底（注意力预算）。

### 四、上下文管理与相邻领域的边界消融

上下文管理正在与三个相邻领域发生边界消融：

- **与长期记忆的融合**：[AI-Native Memory](concepts/AI-Native Memory.md) 模糊了「当前上下文」与「持久记忆」的边界，记忆不再是外部存储，而是上下文的动态延伸

- **与推理优化的耦合**：[Prompt Caching](concepts/Prompt Caching.md) 和前缀复用说明上下文结构直接影响推理效率，上下文设计变成了推理优化的一部分

- **与 Harness 工程的嵌套**：[Harness Engineering](concepts/Harness Engineering.md) 将上下文管理视为 Agent 操作系统的核心子系统，上下文策略成为 Harness 配置的关键参数

## 关键发现

1. **上下文管理的本质不是窗口管理，而是注意力预算分配**——从「装得下」到「值得装」的范式转移，意味着每个 token 都应被视为有限资源

1. **六层范式之间存在「瀑布式降级」规律**——实践中（如 Claude Code 6 层防线），系统总是从最低成本的 L0/L1 开始，仅在压力超过阈值时才触发更重的 L2-L5 策略

1. **「thin control over thick state」揭示了上下文管理的终极架构模式**——控制层尽可能轻量（节省上下文），状态层尽可能持久（外部化存储），这与操作系统的内核/用户空间分离同构

1. **Prompt Caching 正在反向塑造上下文设计**——为了提高缓存命中率，开发者被迫将系统提示、工具定义等固定内容前置，这实际上是「推理基础设施倒逼应用层架构」

1. **多 Agent 架构（L5）并非银弹**——Recursive Language Model 揭示了 orchestrator 轨迹膨胀问题，多 Agent 分治只是将上下文压力从单点转移到了跨 Agent 通信层

## 行动建议

1. **在 OpenClaw 的 Harness 配置中显式建立「上下文预算」机制**——为每个 Agent 设定 token 预算上限，并在 Harness 层实现 L0→L5 的自动降级梯度，参考 Claude Code 的 6 层防线模型

1. **将 Prompt Caching 作为 System Prompt 设计的硬约束**——重构 OpenClaw 九层架构，确保静态层（身份、工具定义、核心规则）严格前置且内容稳定，最大化缓存命中率

1. **在知识 Wiki 编译流程中引入「上下文成本标注」**——为每个 concept 页面标注其注入上下文时的 token 开销，让下游 Agent 在检索时能基于预算做出取舍决策
