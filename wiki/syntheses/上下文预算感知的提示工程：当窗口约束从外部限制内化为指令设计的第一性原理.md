## 研究问题

提示工程（Prompt Engineering）和上下文管理（Context Management）在传统认知中是两个独立学科——前者关注「怎么写指令让模型听懂」，后者关注「怎么管理信息让模型看到」。但在 Agent 时代，两者正在深度融合：**上下文窗口的物理约束正在从提示工程的外部限制，内化为指令设计的第一性原理**。这种融合具体表现为哪些结构性变迁？它如何重新定义「好的提示词」的评判标准？

## 综合分析

### 一、从「写好一句话」到「管好一桌材料」：提示工程的范式跃迁

提示工程正在经历一次根本性的范式跃迁——从关注单次指令的措辞质量，转向关注多轮交互中信息流的系统性编排。

| **维度** | **经典 Prompt Engineering** | **上下文感知 Prompt Engineering** |

| --- | --- | --- |

| 核心关注 | 措辞、角色设定、示例质量 | 信息密度、加载时机、窗口预算分配 |

| 优化目标 | 单次输出质量 | 多轮循环中的累积有效性 |

| 关键约束 | 模型能力边界 | 上下文窗口容量 × Token 成本 |

| 设计单元 | 一段提示词文本 | System Prompt + 按需加载 + 压缩策略的三层架构 |

| 失败模式 | 模型误解意图 | 上下文膨胀（Context Bloat）、上下文税 |

| 代表实践 | Few-Shot、Chain-of-Thought | Context Engineering、SDD 规范驱动开发 |

[Context Engineering](concepts/Context Engineering.md) 的核心洞察正是这一跃迁的缩影：与其让用户手动写 prompt 控制 AI，不如**系统性管理 AI 在每次交互中能「看到」的信息**——让失忆的强力同事每次坐下来时，桌上已经摆好了所有需要的材料。这意味着提示工程的重心从「怎么写」迁移到了「放什么、何时放、放多少」。

### 二、上下文税：提示工程的隐性成本结构

[上下文税](concepts/上下文税.md) 揭示了一个被广泛忽视的结构性问题：为了降低遗漏风险而持续堆叠背景材料、历史对话和冗余说明，导致上下文有效信息密度下降，并额外消耗大量 Token。

这个概念重新定义了提示工程的评判标准：

- **好的提示词不再只是「说清楚了」**，还必须是**「说得够少」**。每一个多余的 token 都在稀释真正有用的信号。

- **上下文税的根源**是缺少上下文裁剪、检索分层和提示模板治理——本质上是提示工程缺少上下文意识。

- 它直接拉低单位 Token 产出，使模型调用成本与时延同步上升，将提示质量问题转化为经济问题。

### 三、Context Bloat：从技能膨胀到注意力稀释的传导链

[Context Bloat](concepts/Context Bloat.md) 进一步揭示了上下文管理失败如何沿着传导链摧毁提示工程的效果：

1. **技能描述膨胀** → 过多 Skill 描述、规则说明、工具清单一次性塞进上下文

1. **信号稀释** → 无关上下文占比过高，真正有用的指令被淹没

1. **决策失真** → 误选 Skill、误调用工具、分支判断混乱

1. **执行退化** → 延迟上升、输出质量下降

这条传导链表明：**提示工程的上限不是由指令质量决定的，而是由上下文管理的质量决定的**。即使写出了完美的系统提示词，如果它被淹没在膨胀的上下文中，效果也会大打折扣。

### 四、System Prompt 工程：提示与上下文的融合界面

[System Prompt 工程](concepts/System Prompt 工程.md) 是提示工程与上下文管理的关键融合界面。它不只是「写一段更长的提示词」，而是设计一套可迁移、可维护的行为约束系统。

融合的三个层次：

- **静态层**：角色设定、输出格式、行为边界——这是经典提示工程的领地

- **动态层**：根据当前任务按需注入相关上下文——这是上下文管理的领地

- **元层**：控制静态层和动态层如何协同工作的策略——这是两者融合后的新领域

[OpenClaw 九层 System Prompt 架构](concepts/OpenClaw 九层 System Prompt 架构.md) 展示了这种融合的工程化实践：将系统提示词分层设计，每一层承载不同粒度的上下文信息，实现提示指令与上下文管理的统一编排。

### 五、上下文窗口：提示工程的物理约束层

[上下文窗口](concepts/上下文窗口.md) 作为 LLM 单次调用时最多能读取的文字范围，是提示工程的硬性物理约束。Agent 设计必须决定哪些信息保留、压缩或丢弃——本质上是容量管理问题。

这个约束催生了一系列「上下文感知」的提示工程模式：

- [Procedural Skill](concepts/Procedural Skill.md)：将复杂流程编码为结构化提示，在有限窗口内最大化行为规约密度

- [Skill File](concepts/Skill File.md)：以文件形式管理可复用的提示模块，按需加载而非全量注入

- [精准上下文挂载](concepts/精准上下文挂载.md)：根据任务类型选择性注入上下文，避免「保险式喂料」

- [Chat Template](concepts/Chat Template.md)：标准化对话结构，减少格式冗余消耗的窗口空间

- [Tool Wrapper 模式](concepts/Tool Wrapper 模式.md)：将工具调用封装为上下文友好的提示结构

### 六、前馈控制与反馈控制：提示设计的控制论视角

[前馈控制](concepts/前馈控制.md) 将控制论引入提示工程，提供了一个统一的分析框架：

- **前馈控制**（提示工程主导）：在 Agent 执行之前，通过精心设计的指令、规范和约束**预先**引导行为方向

- **反馈控制**（上下文管理主导）：在 Agent 执行过程中，根据中间结果动态调整上下文内容和指令策略

两者的协同设计决定了 Agent 的整体行为质量。纯前馈（只设计好指令）会在复杂任务中失效，纯反馈（只根据结果调整）会浪费大量 Token 在试错上。最优策略是**前馈设定基线 + 反馈微调偏差**。

## 关键发现

1. **「读改比」是衡量提示—上下文协同效率的核心指标**：[读改比](concepts/读改比.md) 概念揭示了一个被忽视的效率维度——Agent 读取的上下文量与实际产出的修改量之比。提示工程的目标不应只是「让模型理解意图」，而是**最小化读改比**：用最少的上下文输入产生最精准的输出。这要求提示设计与上下文裁剪必须联合优化。

1. **上下文税将提示质量问题转化为经济问题**：在 Token 按量计费的商业模式下，上下文税直接体现为成本开销。这意味着提示工程不再只是一个工程质量问题，而是一个成本优化问题。「写得好」和「花得少」变成了同一个优化目标。

1. **Context Bloat 是提示工程最大的无声杀手**：一个精心设计的系统提示词可能因为被过多的 Skill 描述和工具清单淹没而完全失效。这意味着**提示工程的上限由上下文管理的质量决定**，而非指令本身的质量。

1. **Context Engineering 正在取代 Prompt Engineering 成为核心能力**：从 Anthropic 提出 Context Engineering 到 Karpathy 强调上下文工程是 Agent 开发的「内功」，行业正在形成共识：管理 AI 能看到什么，比管理 AI 被告知什么更重要。提示工程正在被重新定义为上下文工程的一个子集。

1. **提示模板化与上下文按需加载的融合催生了「编译式提示」范式**：传统提示是解释型的（每次手动组装），新范式是编译型的——通过 Skill File、[CLAUDE.md](http://claude.md/) 等机制将提示逻辑预编译为可复用模块，运行时根据上下文条件选择性加载。这种范式将提示工程从手工活变成了系统工程。

## 行动建议

1. **在 OpenClaw 的 Skill 设计中引入「上下文预算」元数据**：每个 Skill File 应声明其预期消耗的上下文 Token 量和优先级。Agent 运行时据此做加载决策，避免 Context Bloat。这将提示工程从「写好指令」升级为「设计上下文友好的指令模块」。

1. **建立读改比监控仪表盘**：在 Tizer 的内容管线和 Agent 工作流中，持续追踪每次 Agent 调用的读改比。识别读改比异常偏高的环节——这些是上下文税最严重的瓶颈点，也是提示工程优化的最高 ROI 切入口。

1. **将 Context Engineering 作为 Agent 技能评审的第一标准**：在评审社区提交的 Skill 时，除了检查功能正确性，还应检查其上下文效率——是否做到了最小化上下文占用、是否支持按需加载、是否会引发 Context Bloat。这将推动整个生态向「上下文感知」的提示工程范式迁移。
