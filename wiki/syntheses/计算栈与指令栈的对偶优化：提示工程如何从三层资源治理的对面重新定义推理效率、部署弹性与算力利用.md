## 研究问题

已有 synthesis《从推理加速到算力调度》揭示了推理优化、模型部署与算力基础设施之间的三层资源治理闭环（「计算栈」）。但这个分析缺少了一个关键维度：**提示工程作为「指令栈」，从上层信息压缩的方向对下层算力资源施加重大影响**。Token 预算、上下文窗口容量、注意力分配，与 FLOPS 预算、服务容量、硬件利用率之间存在怎样的结构同构？自上而下的信息压缩（提示优化）与自下而上的算力扩展（硬件升级）这两种优化路径之间存在怎样的张力与互补？

## 综合分析

### 一、两个栈的结构同构

推理优化×模型部署×算力基础设施构成的「计算栈」，与提示工程作用于其上的「指令栈」之间存在一组精确的结构对应：

| **维度** | **计算栈（自下而上）** | **指令栈（自上而下）** | **结构对应** |

| --- | --- | --- | --- |

| 核心资源 | FLOPS / 显存 / 带宽 | Token / 上下文窗口 / 注意力 | FLOPS ↔ Token：计算的两种计价单位 |

| 容量约束 | 模型服务实例数 | 上下文窗口大小 | 服务容量 ↔ 信息容量：同时处理的上限 |

| 利用率目标 | GPU 集群利用率 | 信息密度 / 指令遵循率 | 硬件利用率 ↔ 语义利用率：有效输出占总输入的比例 |

| 优化方向 | 自下而上（硬件升级 → 服务扩容 → 推理加速） | 自上而下（提示压缩 → 上下文精简 → 推理降本） | 两个方向的约束优化 |

这种对偶关系的核心洞察是：**提示工程和计算栈分别从信息密度和算力密度两个维度优化同一个目标函数——单位资源的有效输出**。KV 缓存压缩在计算栈侧减少显存占用，Caveman Mode 在指令栈侧减少输出 Token——两者都是在不降低输出质量的前提下压缩资源消耗。

### 二、提示工程如何“向下」重塑推理优化

[思维链](concepts/思维链.md)（Chain-of-Thought）是提示工程与推理优化交叉的核心概念。显式的中间推理步骤能显著提升复杂任务的表现，但代价是**成倍增加的 Token 消耗和推理延迟**。这将提示工程的决策直接平移到了计算栈的三个层级：

- **推理层**：更长的 CoT 意味着更多的 KV 缓存占用、更长的序列长度、更高的计算量。KV 缓存压缩和 Continuous batching 的价值因 CoT 而被放大。

- **部署层**：CoT 远高于普通推理的 Token 耗费让服务计费模型变得复杂——因为推理 Token 的价格比地位大幅上升，路由网关需要将「是否启用 CoT」纳入路由决策。

- **基础设施层**：大规模 CoT 推理的算力需求的不确定性更高（因为输出长度难以预测），这对 GPU 集群的弹性调度提出了更高要求。

[Caveman Mode](concepts/Caveman Mode.md) 代表了提示工程对推理优化的直接干预：通过预先约束回答风格（去掉铺垫、解释、冗余礼貌语），将回复压缩为更短的代码与结论。它主要优化的是**输出 Token**，而非模型推理能力本身。在实践中，这种「信息压缩」可以节省 75% 的 Token 费用——直接降低了推理层的资源压力，但只适合「已经理解上下文、只需要快速执行」的场景。

这两者的对比揭示了提示工程对推理优化的双向影响：思维链向推理层「加压」（更多计算换取更好结果），Caveman Mode 向推理层「减压」（更少输出换取更低成本）。**何时加压、何时减压，是一个需要计算栈和指令栈联合优化的决策问题**。

### 三、提示工程如何“向下」重塑模型部署

提示工程的设计决策直接影响模型部署层的架构选型。当 [CLAUDE.md](http://claude.md/) 或项目级提示配置成为标准实践时，部署层需要处理的不再只是「用户提示 + 模型回复」，而是「项目配置 + 系统提示 + 用户提示 + 工具调用 + 模型回复」的复合流。

这种复合流对模型网关的设计产生了新要求：

- **提示感知路由**：当不同提示策略对推理资源的需求差异巨大时（Caveman Mode 可能只需普通推理，复杂 CoT 任务需要高计算实例），网关层需要根据提示特征做路由决策——不同于传统的基于负载的路由。

- **提示缓存与复用**：项目级提示配置（如 [CLAUDE.md](http://claude.md/)）在同一项目的多次调用中保持不变，这为部署层的提示缓存和预计算提供了机会——类似于推理层的 Prefill-decode 分离技术。

[演化式 Harness 优化](concepts/演化式 Harness 优化.md)从另一个角度连接了提示工程与部署架构：当提示策略本身成为可搜索的变量时，优化过程会生成大量并行的提示变体——每个变体都需要独立评测，这对模型部署层的并行服务能力和计费模型提出了新要求。

### 四、提示工程如何「向下」重塑算力基础设施

提示工程的优化对算力基础设施的影响主要通过两条路径传导：

**路径 A：“压缩即节能”**

Caveman Mode 式的提示优化直接减少 Token 消耗，这种节省在规模化之后会显著影响算力基础设施的规划。如果全局 Token 消耗可以通过提示优化减少 50%，那么等效于将数据中心的有效容量扩大一倍——或者说，提示优化是一种「虚拟算力扩容」。这与 Stargate 式的实体算力扩容形成有趣对比：一个从信息密度端发力，一个从硬件密度端发力。

**路径 B：“智能指令即动态调度”**

当提示工程能够根据任务复杂度动态选择推理策略（简单任务用快速回复，复杂任务启用 CoT）时，它实质上在做的是「算力需求的智能调度」。这种调度能力如果能被基础设施层感知，就可以优化 GPU 集群的资源分配——当系统检测到大量简单提示时，可以释放算力给其他任务；当检测到复杂 CoT 任务浪时，可以预留资源。

### 五、对偶优化的核心矛盾：“加压”还是“减压”？

计算栈和指令栈的对偶关系存在一个核心矛盾：

- **计算栈的优化方向是「给更多算力」**：更快的 GPU、更大的显存、更高的并行度。

- **指令栈的优化方向是「用更少算力」**：更短的提示、更精准的指令、更少的 Token。

但两者并非零和博弈，而是存在**开关性的互补关系**：

| **场景** | **最优策略** | **原因** |

| --- | --- | --- |

| 算力充裕、任务复杂 | 加压：启用深度 CoT，充分利用硬件 | 模型能力是约束，算力开销可以承受 |

| 算力紧张、任务简单 | 减压：用 Caveman Mode 压缩输出 | 模型能力有余，算力是约束 |

| 算力紧张、任务复杂 | 蜗轮：用模型蒸馏换取算力余量，再用于 CoT | 先在计算栈侧释放空间，再在指令栈侧使用 |

| 算力充裕、任务简单 | 分流：用小模型 + 简单提示处理大量低复杂度请求 | 算力有余，可以用更多实例处理更多请求 |

这个矩阵揭示了一个关键洞察：**最优推理策略不是固定的，而是一个关于算力供给与任务复杂度的二维函数**。计算栈和指令栈的联合优化，本质上是在这个二维空间中动态寻找每个任务的全局最优解。

### 六、从「三层治理」到「四轴优化」的范式升级

原有的三层资源治理模型（推理优化 → 模型部署 → 算力基础设施）描述了计算栈内部的自下而上资源流动。引入提示工程作为第四轴后，模型变为：

| **层级** | **计算栈视角** | **指令栈视角** | **联合优化目标** |

| --- | --- | --- | --- |

| L0 推理层 | KV 缓存压缩、Continuous batching | 思维链设计、Caveman Mode | 单请求的 Token/FLOP 最优比 |

| L1 部署层 | PipeLive 在线重构、多模型网关 | 提示感知路由、提示缓存 | 服务级的资源分配最优 |

| L2 基础设施层 | GPU 集群弹性调度 | 提示优化作为“虚拟算力扩容” | 全局算力成本最优 |

## 关键发现

1. **提示工程是计算栈的「对偶优化器」**：计算栈从硬件向上扩展算力供给，提示工程从信息层向下压缩算力需求。两者优化的是同一个目标函数（单位资源的有效输出），但从相反方向发力。在实践中，Caveman Mode 式的提示压缩可以节省 75% 的 Token 费用，等效于将算力基础设施的有效容量扩大数倍。

1. **思维链是指令栈与计算栈的耦合枢纽**：CoT 同时放大了推理层的资源压力（更多 KV 缓存、更长序列）和部署层的计费复杂度（推理 Token 价格跃升）。是否启用 CoT 不应该是纯提示层的决策，而应当纳入计算栈的全局资源调度中。

1. **「何时加压、何时减压」是一个二维优化问题**：最优推理策略是算力供给与任务复杂度的函数。算力充裕时应「加压」（深度 CoT），算力紧张时应「减压」（Caveman Mode），计算栈和指令栈的联合优化本质上是在这个二维空间中动态寻找每个任务的全局最优解。

1. **提示优化是一种「虚拟算力扩容」**：通过提示压缩减少全局 Token 消耗的效果，等效于在不增加硬件的前提下扩大数据中心的有效容量。这种「从信息密度端发力」的策略，与 Stargate 式的「从硬件密度端发力」的策略形成了结构对称的互补关系。

1. **演化式优化将提示工程从「手工调参」升级为「系统搜索」**：EvoForge 式的演化优化让提示策略成为可搜索的变量，这将在部署层产生大量并行评测需求，反过来推动模型网关和算力基础设施的架构升级。

## 行动建议

1. **在 OpenClaw 编译管线中实施「任务复杂度感知的提示策略」**：对于简单的格式化、标签分类任务，采用 Caveman Mode 式的压缩提示来降低 Token 成本；对于复杂的跨标签 synthesis 编译，允许更充分的 CoT 推理以确保分析质量。这种分层提示策略可以在不降低编译质量的前提下显著降低平均 Token 消耗。

1. **将提示优化作为「虚拟算力扩容」纳入成本规划**：在评估是否需要升级模型或增加 API 预算时，先评估提示优化的空间。如果通过提示重构可以减少 50% 的 Token 消耗，其效果等价于算力预算翻倍——且不需要任何硬件投入。

1. **关注「提示感知路由」作为模型网关的下一步演化方向**：当 Tizer 的多模型使用场景中不同任务的提示复杂度差异巨大时，模型路由应从「基于负载」升级为「基于提示特征」——简单提示路由到轻量模型，复杂 CoT 任务路由到高算力实例，最大化全局资源利用率。
