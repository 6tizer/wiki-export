## 研究问题

Coding Agent 在长期开发协作中如何构建有效的记忆系统？当前存在哪些不同的设计范式，它们各自解决什么层级的问题，以及对 Tizer 的工具选型和工作流设计有什么启示？

## 综合分析

### 一、两条设计路线的分化

综合 7 个概念词条和 3 篇摘要，Coding Agent 记忆系统目前呈现出**两条清晰但互补的设计路线**：

- **路线 A：成本优先的层级压缩**（以 Claude Code 为代表）——核心哲学是"能不总结就不总结"，通过多层递进机制把上下文膨胀拦截在最低成本层。

- **路线 B：协议驱动的结构化记忆**（以 MemPalace / mempal 为代表）——核心哲学是"记忆是基础设施"，通过形式化协议让记忆可验证、可迁移、可跨项目协作。

| **对比维度** | **路线 A：层级压缩（Claude Code）** | **路线 B：协议驱动（MemPalace/mempal）** |

| --- | --- | --- |

| **核心目标** | 控制上下文膨胀成本，维持长会话质量 | 构建可迁移、可验证的长期记忆基础设施 |

| **记忆粒度** | 消息级 → 工具结果级 → 会话级 → 跨会话级 | 概念级 → 项目级 → 跨项目级 |

| **关键机制** | 微压缩、会话记忆、Dreaming 后台整理 | AAAK 方言、混合检索、知识图谱层、跨项目隧道 |

| **存储形式** | 纯文本（[CLAUDE.md](http://claude.md/) / [MEMORY.md](http://memory.md/)） | 结构化协议（Wing/Room/Tunnel 层级） |

| **跨项目能力** | 有限（层级化 md 文件可手动复用） | 原生支持（跨项目隧道自动发现关联） |

| **可验证性** | 无形式化验证机制 | 目标是形式文法 + 往返一致性（仍在建设中） |

| **成熟度** | 生产级（Claude Code 已大规模使用） | 实验级（Rust 重写中，核心协议仍需完善） |

### 二、Claude Code 的「七层记忆金字塔」

两篇独立分析文章从不同角度解构了 Claude Code 的记忆体系，综合来看可以归纳为一个**成本递增的处理金字塔**：

1. **Prompt Cache**（最低成本）— 缓存命中优先，避免重复计算

1. **微压缩** — 删除过时的大型工具结果，轻量清理

1. **会话记忆** — 实时维护结构化笔记，把临时上下文转为可重用中间层

1. **完整压缩** — 逼近上限时才做整段摘要

1. **文档记忆** — 用户主动记录的 [CLAUDE.md](http://claude.md/) 偏好文件

1. **自动记忆** — AI 自动捕获对话中关键偏好（[MEMORY.md](http://memory.md/)）

1. **Dreaming** — 后台异步跨会话整理与巩固

这套设计的精髓在于：**每一层都是对上一层的延迟触发**。微压缩拦截了大部分膨胀，会话记忆再收纳一层，只有真正逼近上限时才启动昂贵的完整压缩。这种"懒处理"哲学直接影响了商业成本和用户体验。

### 三、MemPalace 路线的「协议优先」哲学

MemPalace 及其 Rust 重写版 mempal 走了完全不同的路。它不关心单次会话的上下文管理，而是把注意力放在：

- **AAAK 方言**：一种压缩表达记忆内容的结构化表示。但原实现的关键缺口是缺少形式化文法——没有编码器/解码器和往返一致性验证，AAAK 更像启发式压缩而非严格协议。

- **跨项目隧道**：当同一主题出现在多个项目中时，系统自动发现关联并提示链接。这是 Claude Code 目前没有的能力。

- **自描述协议**：将 MEMORY_PROTOCOL 写入 ServerInfo，让多 Agent 可以低门槛接入同一记忆系统。

### 四、记忆生命周期的三阶段模型

跨所有概念综合来看，Coding Agent 记忆系统正在沿三个阶段演进：

| **阶段** | **核心问题** | **代表机制** | **当前状态** |

| --- | --- | --- | --- |

| **1. 存下来** | 如何在不爆上下文的前提下保留关键信息 | 微压缩、会话记忆、Prompt Cache | ✅ 已成熟 |

| **2. 整理好** | 如何把碎片化记忆巩固为稳定的长期知识 | Dreaming 记忆整合、文档/自动记忆 | ⚡ 快速演进中 |

| **3. 用起来** | 如何让记忆跨项目迁移、被多 Agent 共享 | AAAK 方言、跨项目隧道、自描述协议 | 🔬 实验探索中 |

## 关键发现

1. **"懒处理"是当前最成功的记忆策略**：Claude Code 证明了"能不总结就不总结"比"记住一切"更有效。这一洞察可以直接迁移到任何需要管理长上下文的 Agent 设计中。

1. **Dreaming 机制标志着 Agent 从工具到系统的跃迁**：后台异步整理记忆，是 Coding Agent 开始具备"自主维护"能力的关键信号——它不再只是被动回答问题，而是主动管理自己的知识状态。

1. **跨项目记忆是下一个差异化战场**：Claude Code 在单项目内的记忆已经非常成熟，但跨项目经验迁移仍是空白。MemPalace 的跨项目隧道虽然不成熟，但指出了正确的方向。

1. **记忆协议的可验证性决定多 Agent 协作天花板**：AAAK 方言的教训说明，没有形式化文法的压缩表示在单 Agent 场景下可以凑合，但一旦需要多 Agent 共享记忆，可验证性就成为硬门槛。

1. **两条路线最终会融合**：层级压缩解决"存下来"和"整理好"，协议驱动解决"用起来"——完整的 Coding Agent 记忆系统需要同时具备这两种能力。

## 行动建议

1. **在 OpenClaw 的 Agent 设计中采纳"成本递增金字塔"模式**：为 OpenClaw Agent 的上下文管理实现类似微压缩 → 会话记忆 → 完整压缩的分层策略，优先用低成本清理代替昂贵总结，这对控制 API 成本和维持长任务质量都有直接收益。

1. **关注跨项目记忆迁移的工程化路径**：Tizer 同时在多个项目（OpenClaw、知识 Wiki、内容管线）间工作，目前这些项目的 Agent 记忆是孤立的。可以参考跨项目隧道的思路，在 HITL 工作流中设计一个轻量级的"经验桥接"机制，让一个项目中验证过的模式可以被另一个项目检索和复用。

1. **为知识 Wiki 的编译管线引入 Dreaming 式后台整理**：当前编译器是触发式运行，可以增加一个低优先级的后台任务，定期回顾和整合相近概念、合并冗余条目、标记过时内容——这本质上就是把 Claude Code 的 Dreaming 机制迁移到知识管理场景。
