## 研究问题

当 Agent 获得越来越多的外部技能和工具接入能力时，安全风险如何从「模型层」向「技能层」迁移？现有的安全机制能否构成完整的防御体系？攻击者的利用路径和防御者的应对策略之间存在怎样的结构性博弈？

## 综合分析

### 一、风险迁移：从模型安全到技能安全

传统 LLM 安全关注的是模型本身的对齐和越狱问题。但随着 Agent 技能生态的扩展，安全焦点正在向上迁移：

| 安全层级 | 传统关注点 | 技能时代新风险 | 本次涉及概念 |

| --- | --- | --- | --- |

| **供应链层** | 模型权重篡改 | 恶意技能包注入、代码级后门 | skill-security-auditor |

| **凭证层** | API Key 泄露 | OAuth 过宽授权、凭证横向移动 | Vaults、OAuth 授权、OAuth Scope 最小化 |

| **执行层** | 代码注入 | 沙箱逃逸、动态代码提权 | 代码执行沙箱 |

| **运行时信任层** | 提示注入 | 混淆代理人攻击、数据窃取链路 | CaMeL 信任边界、数据窃取陷阱、AI Crawl Control |

这四个层级不是独立的——它们形成了一条**攻击放大链**：恶意技能包（供应链）→ 获取过宽凭证（凭证层）→ 在沙箱中执行恶意代码（执行层）→ 利用 Agent 信任将数据外泄（运行时）。

### 二、供应链层：技能安装前的守门机制

skill-security-auditor 代表了一种「安装前审计」思路：在 Agent 加载新技能包之前，先用专门的审计型技能扫描恶意代码和提示注入。这与传统软件的包管理安全（npm audit、dependabot）高度类似。

**关键问题**：谁来审计审计者？如果 skill-security-auditor 本身被篡改，整条供应链防线就会失效。这要求技能生态必须有**可信根**——类似代码签名或可验证构建的机制。

AI Crawl Control 则从另一个角度切入供应链安全：当 Agent 技能包括网页抓取能力时，被抓取方需要声明访问控制策略。这体现了一个新趋势——**安全治理不只是 Agent 持有者的责任，还需要被访问方的协同参与**。

### 三、凭证层：Vercel 事件揭示的 OAuth 放大效应

Vaults、Google Workspace OAuth 授权、OAuth Scope 最小化三个概念共同描绘了凭证管理的攻防图景。Vercel 安全事件是一个典型案例：

| 攻击阶段 | 利用机制 | 对应防御 |

| --- | --- | --- |

| 初始入侵 | 第三方 AI 工具获得过宽 OAuth 授权 | OAuth Scope 最小化 |

| 横向移动 | 利用 Google Workspace 授权访问内部资源 | Vaults 凭证隔离 |

| 数据外泄 | 通过持续访问能力持续窃取数据 | 周期复核 + 可撤销性治理 |

Anthopic 的 Vaults 机制代表了平台级解决方案：把外部服务密钥与模型上下文隔离，Agent 能使用凭证但无法读取凭证内容。这与传统 KMS（密钥管理系统）的设计哲学一致，但在 Agent 语境下面临新挑战——**Agent 需要动态获取和释放凭证，而传统 KMS 假设调用者身份固定**。

### 四、执行层：沙箱隔离的层级光谱

代码执行沙箱是 Agent 运行动态代码的基本安全保障。但「沙箱」这个概念跨越了多个隔离层级：

- **进程级**：V8 Isolate（Cloudflare Dynamic Workers），启动快、资源占用低，但隔离强度有限

- **容器级**：Docker / OCI 容器，平衡隔离与性能

- **微虚拟机级**：Firecracker / gVisor，接近硬件级隔离

关键权衡是**隔离强度 vs 冷启动延迟**。对于 Agent 工作流中频繁调用的轻量技能（如数据转换、格式处理），V8 Isolate 级别的快速隔离更实用；对于涉及网络访问和文件操作的高风险技能，则需要容器级甚至微虚拟机级的隔离。

### 五、运行时信任层：CaMeL 架构的范式突破

CaMeL 信任边界和数据窃取陷阱共同揭示了 Agent 安全中最深层的问题：**当 Agent 同时处理可信指令和不可信数据时，如何防止数据污染执行链路？**

CaMeL 的核心思想是把 Agent 运行时分为两个层面：

- **可信控制层**：系统指令、用户意图、已批准的能力集

- **不可信数据层**：网页内容、工具输出、检索结果、会话回忆

敏感操作必须对照可信计划授权，而不是直接听从不可信内容中的指令。这是对传统「提示注入防御」的架构级升级——不再试图让模型区分善恶指令，而是从**系统架构上隔离信任域**。

数据窃取陷阱则展示了攻击者如何利用 Agent 的「混淆代理人」角色：通过在网页、邮件或文档中嵌入伪装成正常指令的恶意内容，引导 Agent 把用户机密信息发送到攻击者端点。这种攻击之所以危险，正是因为 Agent 已经拥有了读取私密数据和调用外发工具的技能。

### 六、四层防线的协同与缺口

| 防御层 | 当前成熟度 | 最大缺口 |

| --- | --- | --- |

| 供应链审计 | ⭐⭐ 早期 | 缺乏可信根和持续监控 |

| 凭证管理 | ⭐⭐⭐ 平台级方案已出现 | 动态凭证生命周期管理 |

| 执行隔离 | ⭐⭐⭐⭐ 基础设施成熟 | 隔离层级自适应选择 |

| 运行时信任 | ⭐⭐ 理论框架刚建立 | 落地工程化与性能开销 |

## 关键发现

1. **Agent 安全的核心矛盾是「能力越强，攻击面越大」**：每新增一个技能（邮件读取、文件操作、网页浏览），就同时增加了一个潜在的数据窃取通道。安全不能事后补救，必须在技能注册时就完成权限声明和风险评估

1. **OAuth 不是登录动作，而是安全治理对象**：Vercel 事件证明，给 AI 工具的 OAuth 授权等同于给予持续的企业内部访问权。Scope 最小化不是可选的最佳实践，而是必须的安全约束

1. **CaMeL 信任边界代表了从「检测恶意」到「隔离信任域」的范式转换**：不再尝试让模型分辨善恶指令，而是从架构上确保不可信数据无法直接触发敏感操作。这是目前最有前景的 Agent 安全方向

1. **供应链安全是当前最薄弱的环节**：技能市场缺乏代码签名、可验证构建和持续监控机制。skill-security-auditor 是一个好的起点，但距离完整的供应链安全体系还有很大差距

1. **被访问方的安全协同正在成为新范式**：AI Crawl Control 表明，Agent 安全不只是 Agent 持有者的责任——被抓取的网站、被调用的 API、被访问的文档都需要参与安全治理

## 行动建议

1. **在 OpenClaw 的技能注册流程中强制引入安全审计环节**：参考 skill-security-auditor 的思路，在技能安装前扫描恶意代码、提示注入和过宽权限声明。长期应推动技能包签名和可验证构建

1. **为 Agent 工作流中的外部服务接入实施 OAuth Scope 最小化策略**：尤其是 Google Workspace、GitHub 等高价值目标。建议为 OpenClaw 的凭证管理模块引入类似 Vaults 的隔离机制，确保 Agent 能使用但不能读取凭证

1. **关注 CaMeL 信任边界的工程化落地**：这是目前最有前景的运行时安全方向。建议在 HITL 工作流中率先实验信任域隔离——把用户审批作为可信控制层的一部分，确保高风险操作不会被不可信数据触发
