## 研究问题

链上隐私计算的核心密码学技术（ZKP、MPC、FHE、TEE）各自解决什么问题？它们如何组合形成完整的隐私基础设施栈？这些技术与 AI Agent 的交叉点在哪里，对 Tizer 的 Crypto/DeFi 实践意味着什么？

## 综合分析

### 一、四大密码学原语的能力边界

从 Wiki 中的 7 个交叉概念出发，可以将链上隐私计算的核心技术分为四大原语，它们各自有明确的能力边界和适用场景：

| **技术原语** | **核心能力** | **解决的问题** | **主要瓶颈** | **Wiki 概念** |

| --- | --- | --- | --- | --- |

| 零知识证明（ZKP） | 证明「知道某事」而不泄露内容 | 链上计算可信性验证 | 证明生成计算开销大 | [Untitled](concepts/零知识证明.md) |

| 多方计算（MPC） | 多方联合计算，不暴露各自输入 | 多主体协作下的隐私保护 | 通信复杂度随参与方增长 | [Untitled](concepts/多方计算.md) |

| 全同态加密（FHE） | 直接在密文上计算 | 敏感数据处理（借贷、医疗、信用） | 计算成本极高，工程不成熟 | [Untitled](concepts/全同态加密.md) |

| 可信执行环境（TEE） | 硬件隔离的安全计算区域 | 高性能隐私计算的工程折中 | 依赖硬件厂商信任假设 | （关联概念，多页引用） |

**关键洞察**：这四种技术不是竞争关系，而是分层互补。ZKP 解决「验证」问题，MPC 解决「协作」问题，FHE 解决「计算」问题，TEE 解决「性能」问题。

### 二、Privacy 2.0 的演进框架

[Privacy 2.0](concepts/Privacy 2.0.md) 概念提供了理解链上隐私演进的最佳框架：

| **阶段** | **隐私模型** | **典型方案** | **局限性** |

| --- | --- | --- | --- |

| Privacy 0.9 | 交易混淆 | Mixer（Tornado Cash 等） | 仅隐藏资金流向，无法支持复杂计算 |

| Privacy 1.0 | 独立隐私链 | Zcash、Monero | 生态孤岛，无法与主流 DeFi 组合 |

| Privacy 2.0 | 可编程、可组合、可合规的隐私能力 | Nillion、选择性披露协议 | 技术组合复杂度高，标准未统一 |

Privacy 2.0 的核心范式转变在于：隐私不再是「全有或全无」的二元选择，而是围绕**匿名性、机密性、不可关联性、选择性披露**四个维度做精细化产品设计。[选择性披露](concepts/选择性披露.md)的理念尤为关键——未来主流方向更可能是「可验证但不裸奔」的合规可审计机密性，而不是完全匿名。

### 三、Nillion 盲计算：技术组合的集大成者

[Nillion 盲计算](concepts/Nillion 盲计算.md)是当前最具野心的技术整合方案，它同时组合了 MPC + FHE + TEE + ZKP 四种技术：

- **Curl 协议**：Nillion 自研 MPC 协议，支持不解密条件下的复杂非线性计算

- **双层架构**：Nilchain（协调层）负责交易和请求路由，PETnet（隐私增强技术网络）多集群节点执行盲计算

- **三大产品模块**：

  - nilDB：加密数据库（数据始终以密文存储）

  - nilVM：分布式计算引擎（在加密数据上执行计算）

  - nilAI：隐私 AI 推理（模型在不见明文的情况下推理）

**nilAI 模块是与 AI Agent 最直接的交叉点**：它意味着 Agent 可以在不接触用户原始数据的情况下提供个性化服务，从根本上解决了「个性化 vs 隐私」的矛盾。

### 四、EIP-7702 与链上 Agent 的安全执行层

[EIP-7702](concepts/EIP-7702.md) 从另一个维度介入安全问题——不是密码学层面的隐私保护，而是账户层面的权限控制：

- 让普通 EOA 账户临时委托给合约逻辑执行

- 使 Gas 赞助和批量执行等能力无需升级到完整智能账户

- 对 Agent 自动执行交易场景尤为关键：Agent 可以获得受限的执行权限，而不是完全控制用户钱包

这与 SlowMist 提出的 AI Agent 三层防御矩阵（事前确认、事中拦截、事后巡检）形成了技术互补：密码学技术保护数据隐私，账户抽象保护资产安全。

### 五、技术栈组合矩阵

将所有概念整合，可以画出一个从底层到应用的技术栈分层：

| **层次** | **技术 / 协议** | **AI Agent 应用场景** |

| --- | --- | --- |

| 密码学原语层 | ZKP、MPC、FHE、TEE | 链上 AI 推理可信性证明、多 Agent 联合计算 |

| 隐私计算中间件 | Nillion（nilDB + nilVM + nilAI） | Agent 隐私训练、个性化推理 |

| 账户与执行层 | EIP-7702、Paymaster、智能账户 | Agent 自动交易、权限最小化执行 |

| 合规与披露层 | 选择性披露、Privacy 2.0 协议 | Agent 链上行为审计、合规交互 |

## 关键发现

1. **四大密码学原语是「分层互补」而非「互相替代」**：ZKP 验证、MPC 协作、FHE 计算、TEE 加速——每种技术解决不同层面的问题。真正的隐私基础设施（如 Nillion）需要同时组合多种技术。单独押注任何一种都不够

1. **Privacy 2.0 最反直觉的洞察：「选择性披露」比「完全匿名」更有商业价值**。合规可审计的机密性才是机构采用的关键桥梁。这意味着面向 B 端的 Agent 隐私方案应该优先实现「谁能看见什么、在什么条件下看见」，而不是追求绝对匿名

1. **Nillion 的 nilAI 可能是解决 AI Agent 个性化与隐私矛盾的第一个可行方案**：以往 Agent 要提供个性化服务就必须看到用户数据。nilAI 让 AI 模型在不见明文的情况下推理，这是一个范式级的变化。对 HITL（Human-in-the-Loop）工作流的影响尤其深远

1. **EIP-7702 揭示了一个被忽视的安全维度**：大多数关于 Agent 安全的讨论聚焦在提示注入和数据隐私，但「链上 Agent 的执行权限管控」同样关键。临时委托 + 权限最小化是链上 Agent 安全的实用工程方案

1. **密码学技术正在从「区块链基础设施」扩展为「AI Agent 基础设施」**：ZKP 可证明 AI 推理正确性、MPC 可实现多 Agent 隐私协作、FHE 可支持隐私训练——这些技术的 AI 应用场景可能比区块链原生场景更有增长潜力

## 行动建议

1. **在 OpenClaw Agent 的链上交互模块中优先接入 EIP-7702 的临时委托机制**：不要让 Agent 直接持有完整钱包私钥。通过临时委托 + Paymaster 赞助的组合，实现权限最小化的自动交易执行，这是当前工程成本最低、安全收益最高的改进

1. **持续跟踪 Nillion nilAI 的开发进展，评估将其作为 HITL 工作流隐私层的可行性**：如果 nilAI 成熟，用户的个人数据（交易偏好、风控参数等）可以加密方式参与 Agent 训练和推理，从根本上解决 OpenClaw 在处理用户敏感数据时的隐私合规问题

1. **将选择性披露理念应用到 Agent 的链上行为审计系统设计中**：为未来的链上 Agent 设计「可验证但不裸奔」的行为日志——交易对手可以验证 Agent 的执行逻辑正确性（通过 ZKP），但无法看到具体的策略参数和持仓细节
