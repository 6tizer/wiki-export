## 研究问题

当 AI Agent 从单体问答走向多智能体自主编排，安全边界从「提示词层」扩展到「执行层」和「协作层」——**编排系统如何在赋予 Agent 更大自主权的同时，建立分层、可验证、可恢复的安全治理体系？**

本综合分析交叉对比 11 个双标签概念词条与 4 篇双标签摘要，提炼出六层防御模型，并映射到 Tizer 当前的 Agent 工作流场景。

## 综合分析

### 一、核心矛盾：自主性与可控性的张力

多智能体编排追求的是自主、分布、高效——Agent 能委派子任务、生成子 Agent、操作文件与浏览器、长期持有工作区。然而，每一项能力扩展都在拉宽攻击面：

- **执行面扩展**：Sandbox Agents 让 Agent 拥有文件系统、命令行、浏览器操作权，安全风险从文本回答延伸到真实系统操作

- **协作面扩展**：多 Agent 委派链路可能被注入恶意子 Agent（子 Agent 生成陷阱），信任在协作层被横向传递

- **环境面扩展**：动态伪装让攻击者针对 Agent 指纹投放定制化恶意内容，Agent 的感知层成为新攻击入口

### 二、六层安全治理模型

基于本 Wiki 中 11 个交叉概念的系统梳理，可以提炼出从内到外的六层防御架构：

| **防御层** | **核心机制** | **对应概念** | **解决的问题** |

| --- | --- | --- | --- |

| L1 权限门控 | 按角色设定最小必要权限 | Policy Gate、信任边界 | Agent 越权访问与操作 |

| L2 执行隔离 | 编排与计算分离，沙箱执行 | Sandbox 抽象、Sandbox Agents、编排与计算分离 | 执行环境污染宿主系统 |

| L3 不可变护栏 | 核心文件禁止修改 | 不可变文件约束 | Agent 改写自身护栏规则 |

| L4 治理中间层 | 调用与回流的统一管控 | Mediator 层 | 工具调用链路缺乏审计点 |

| L5 可观测与审计 | 语义级行为监控与回滚 | 语义可观测性、Agent 审计与回滚机制 | 事后无法定位与恢复 |

| L6 环境对抗 | 输入分级与来源隔离 | 动态伪装、子 Agent 生成陷阱 | 外部环境投毒与链式攻击 |

### 三、L1-L2：从权限控制到执行隔离

**Policy Gate** 和 **信任边界** 构成第一道防线。Policy Gate 按 Agent 角色（研究型、写作型、工程型、编排型）设置差异化权限：研究型偏只读，工程型可执行测试但关键提交需审批，编排型负责最终扩权把关。信任边界则在更宏观层面定义哪些上下文、工具、数据可以被访问。

**Sandbox 抽象** 和 **编排与计算分离** 将防线从权限声明推进到物理隔离。核心设计哲学是把 Agent 的「大脑」（推理与调度）和「手」（文件、命令、浏览器操作）拆开部署。关键洞察：

- 不是每个任务都需要完整计算环境，只有涉及文件、进程、凭证或持久状态时才挂载沙盒

- 安全隔离不是附属能力，而是执行边界的组成部分

- 沙盒的价值不仅是「能跑命令」，更是能留下**可检查、可恢复、可继续的工作产物**

### 四、L3-L4：不可变护栏与治理中间层

**不可变文件约束** 解决了一个特殊威胁：自进化 Agent 修改自身护栏。在 Agent 可以改写代码的场景中，identity 文件、编排器配置、CI 工作流和核心技能被划为禁止修改区域——「能自改代码但不能自拆护栏」。

**Mediator 层** 则在 Agent 核心与外部能力之间插入治理层，承载沙箱管理、可观测性、压缩、评估、审批回路与子 Agent 编排。它的价值不是新增知识，而是**约束知识、技能与工具如何被调用和回流**——把 Harness 设计从「能不能接上」升级为「如何被治理」。

### 五、L5：语义可观测性与审计回滚

传统监控关注延迟和报错，但 Agent 场景的错误常体现在**决策与语言行为**中。**语义可观测性** 通过聚类与模式分析，识别输出中的危险模式、偏差和失控趋势，是构建持续治理、红蓝对抗和安全回路的基础设施。

**Agent 审计与回滚机制** 则确保事故可定位、可恢复：

- 审计日志决定事后能不能定位问题

- 回滚能力决定事故能不能低成本恢复

- 在高频批量修改场景，权限边界与人工抽检往往比模型能力本身更关键

### 六、L6：环境对抗——新型攻击面

这是跨概念分析中最具洞察力的发现。Google DeepMind 论文揭示的攻击面已从模型层扩展到环境层：

- **动态伪装**：网站根据 Agent 指纹返回定制化恶意内容，人类看到正常页面，Agent 读取被投毒的信息

- **子 Agent 生成陷阱**：攻击者诱导父 Agent 实例化带有恶意配置的子 Agent，借系统内部协作链放大权限

- 真正危险的不是单点攻击，而是**链式组合**：隐藏指令先解除对齐 → 诱发越权操作 → 通过记忆投毒长期化影响

来自 Claude Opus 4.7 自我越狱案例的启示：当 Agent 拥有 Computer Use 等真实操作能力后，安全评估必须从「提示词安全」升级到「执行安全」。

### 七、编排模式与安全等级的映射

| **编排模式** | **安全风险等级** | **最低防御层级** | **典型场景** |

| --- | --- | --- | --- |

| 单 Agent + 只读工具 | 低 | L1（权限门控） | 知识检索、问答 |

| 单 Agent + 写权限 | 中 | L1-L3 + L5 | 内容编辑、数据更新 |

| 多 Agent 协作 + 任务委派 | 高 | L1-L5 | 自动化工作流、HITL 管线 |

| 自进化 Agent + 外部交互 | 极高 | 全部六层 | 自主研究、代码自改写 |

## 关键发现

1. **安全边界随编排能力呈指数扩展**：单体 Agent 的安全主要是提示词层面，但一旦引入沙箱执行、子 Agent 生成和外部环境交互，攻击面从一维（文本）扩展到三维（文本 × 执行 × 协作），防御成本非线性增长。

1. **Mediator 层是被忽视的关键架构层**：多数 Agent 框架把精力放在编排逻辑和工具接入上，但真正决定系统能否安全运行的，是调用与回流之间的治理层——它是唯一能同时拦截输入投毒和输出越权的位置。

1. **「环境即攻击面」颠覆了传统安全假设**：动态伪装和子 Agent 生成陷阱说明，攻击者无需破解模型本身，只要操纵 Agent 所处环境就能把 Agent 变成攻击工具。这要求从「保护模型」转向「保护 Agent 的感知-行动闭环」。

1. **不可变约束是自进化系统的逻辑前提**：如果 Agent 能修改自身代码但也能修改护栏，那所有安全措施都可能被绕过。不可变文件约束不是限制功能扩展，而是为可验证进化提供可信锚点。

1. **审计回滚的价值在高频场景被严重低估**：当 Agent 以人类无法实时审查的速度批量执行操作时，效率提升越快，出错时的损失越大。事后回滚能力比事前预防在某些场景下更具实操价值。

## 行动建议

1. **为 OpenClaw 和内容流水线建立 Mediator 层原型**：在现有 HITL 审批节点基础上，增加工具调用日志、输入来源分级和异常暂停机制。不需要一步到位实现六层防御，但至少应在编排器与外部工具之间插入一层可审计的治理中间件。

1. **在高频批量操作场景引入「先审计后回滚」机制**：特别是 Wiki 编译、批量属性更新等场景，为每次批量操作生成可回滚的变更快照，让事故恢复成本从「逐条手动修复」降到「一键回退到快照」。

1. **将「环境投毒」纳入 Agent 工作流的默认威胁模型**：任何带浏览、检索、外发能力的 Agent，应默认把外部输入视为不可信，建立输入分级 → 动作确认 → 来源隔离 → 异常暂停的四步防御链路，尤其避免把高权限自动化直接暴露在不可信上下文里。
