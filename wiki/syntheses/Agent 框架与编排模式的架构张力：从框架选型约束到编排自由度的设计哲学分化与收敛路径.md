## 研究问题

Agent 框架（底层基础设施）与编排模式（上层调度逻辑）之间存在怎样的架构张力？框架的设计哲学如何约束和塑造编排范式的选择空间？当框架从应用层胶水代码进化到操作系统级基础设施时，编排模式的自由度和复杂度如何随之演变？

本文综合 21 个横跨 Agent 框架与 Agent 编排的 concept/entity 条目及相关摘要，试图回答：**框架选型决策如何从根本上决定编排架构的天花板。**

## 综合分析

### 一、框架-编排关系的四种架构范式

从已有条目中，可以提炼出框架与编排之间四种递进的耦合范式：

| **范式** | **框架定位** | **编排特征** | **代表项目** | **设计哲学** |

| --- | --- | --- | --- | --- |

| 胶水层框架 | 应用层连接器 | 编排逻辑写在应用代码中 | AG2、传统 LangChain | 框架不管编排，开发者自行实现 |

| 编排优先框架 | 内置多种编排模式 | 框架提供预定义编排模板 | Swarms（5 种编排模式）、Agentic Engine | 框架即编排，开发者选模板 |

| 方法论驱动框架 | 软件工程方法论的代码化 | 角色分工 + 流程衔接 | BMAD Method（敏捷团队模拟）、AI Hedge Fund（投资大师编排） | 编排即组织架构，框架是组织的骨架 |

| 操作系统级框架 | 内核级调度与资源管理 | Agent 作为进程，编排作为调度策略 | Agent OS / OpenFang（Hands 机制）、SuperAgent Harness | 框架即操作系统，编排是内核调度 |

### 二、关键架构转折：从「框架提供编排」到「框架约束编排」

传统理解中，框架是编排的工具——开发者用框架来实现编排逻辑。但当框架进化到操作系统级时，关系发生了根本反转：

- **Agent OS 的 Hands 机制**将 Agent 从被动响应变为主动执行的进程。编排不再是「谁调用谁」的问题，而是「操作系统如何调度多个持续运行的进程」。这意味着编排模式被框架的进程模型所约束——你无法在一个只支持请求-响应的框架上实现真正的自主调度。

- **OpenFang 的 16 层安全系统**（WASM 沙箱、Ed25519 签名、Merkle 审计链）为编排设定了安全边界。每个 Agent 的能力范围在编译时就被确定，编排逻辑必须在这些安全约束内运行。

- **插件化架构**（以 Dify v1.0 为典型）提供了一种中间路径：框架提供标准化的扩展接口，编排逻辑通过插件组合实现，而非硬编码在框架中。

### 三、五种框架对编排能力的约束维度

| **约束维度** | **胶水层框架** | **编排优先框架** | **方法论框架** | **操作系统级框架** |

| --- | --- | --- | --- | --- |

| 并发能力 | 开发者自行实现 | 内置并发工作流 | 角色并行但受流程约束 | 内核级进程调度 |

| 状态管理 | 无内置状态 | 框架内状态共享 | 角色间交接文档 | 持久化进程状态 |

| 安全隔离 | 无隔离 | 基础权限控制 | 角色边界隔离 | WASM 沙箱 + 资源计量 |

| 编排灵活度 | 最高（无约束） | 模板内灵活 | 方法论规定流程 | 调度策略可配置 |

| 部署复杂度 | 最低 | 中等 | 高（角色配置多） | 最高（系统级部署） |

### 四、子 Agent 派生：框架与编排融合的前沿样本

**子 Agent 派生**（OpenClaw 2026.2.17 引入）代表了一种值得关注的趋势——框架直接提供动态编排原语：

- 主 Agent 在运行时根据需要创建子 Agent，无需预定义编排图

- 配合 1M 上下文窗口，子 Agent 之间的交接文档可以无限长

- 编排从「设计时确定」变为「运行时涌现」

这种模式模糊了框架与编排的边界——框架提供的是编排能力的「原子操作」，而非完整的编排方案。实际编排模式由 Agent 在运行时自主决定。

### 五、Deep Agents Deploy 与 SuperAgent Harness：生产化的架构收敛

无论框架的设计哲学如何不同，走向生产部署时都面临相同的收敛压力：

- **Deep Agents Deploy**（LangChain）将多租户编排、沙箱和协议接入封装为一条命令部署

- **SuperAgent Harness** 将模型、工具、记忆、执行环境和任务编排整合为统一运行时

这表明：**框架层面的分化最终会在生产部署层面收敛**。无论你选择了哪种框架哲学，到了生产环境，都需要解决调度、隔离、记忆和可观测性这四个基础问题。

### 六、多步骤研究规划：编排智能内化到框架的信号

**多步骤研究规划**揭示了一个重要趋势：编排智能正在从显式的外部定义，转向内化为框架的内置能力。当 Agent 具备自主规划多步骤研究路径、动态调整策略、支持回溯的能力时，传统的编排器（orchestrator）角色就被部分取代了。这意味着框架的价值正在从「提供编排工具」转向「提供自主规划基础设施」。

## 关键发现

> **💡** **发现 1：框架的抽象层级决定编排的天花板。** 胶水层框架能实现任意编排逻辑但无安全保障；操作系统级框架提供强安全保障但编排必须在其约束内运行。这不是缺陷，而是有意的设计取舍——安全性和灵活性在框架层面就已经做了权衡。

> **💡** **发现 2：「框架即编排」和「框架约束编排」是两种根本不同的设计哲学，但正在走向融合。** Swarms 的 5 种内置编排模式代表「框架即编排」，OpenFang 的 Hands 机制代表「框架约束编排」。子 Agent 派生则介于两者之间——框架提供原语，编排在运行时涌现。这条融合路径可能是未来主流。

> **💡** **发现 3：方法论驱动的框架（BMAD Method）揭示了一个被忽视的维度——编排不仅是技术问题，更是组织设计问题。** 当框架将产品经理、架构师、QA 等角色代码化时，它实际上是在模拟一个软件组织的协作结构。框架的设计质量等同于组织设计的质量。

> **💡** **发现 4：生产部署是框架分化的终极收敛点。** 无论框架哲学如何不同，Deep Agents Deploy 和 SuperAgent Harness 都指向同一结论：调度、隔离、记忆、可观测性是必须解决的四个基础问题。框架选型的真正标准不是「编排能力多强」，而是「这四个基础问题解决得多好」。

> **💡** **发现 5：编排智能正在从外部编排器迁移到框架内部。** 多步骤研究规划和 autoresearch 的自主实验循环表明，Agent 越来越不需要外部编排器来告诉它「下一步做什么」。这对框架设计的启示是：与其提供更多编排模板，不如提供更好的自主规划基础设施。

## 行动建议

1. **在 OpenClaw Agent 开发中优先验证「子 Agent 派生 + 自主规划」模式**：这是框架-编排融合的前沿范式。在知识 Wiki 编译管线中实验：让 Compiler Agent 在运行时根据文章复杂度自主决定是否派生子 Agent 分担工作，而非预定义固定的编排流程。这将验证「运行时涌现编排」在实际知识工作中的可行性。

1. **建立框架选型的四维评估矩阵**：基于本文发现的「调度、隔离、记忆、可观测性」四个收敛维度，为 Tizer 的 Agent 基础设施选型建立标准化评估框架。当评估 OpenFang、Swarms、LangGraph 等方案时，不看编排模板数量，而看这四个维度的成熟度。

1. **将 BMAD Method 的组织模拟思路应用于内容生产团队**：把知识 Wiki 的编译、审核、综合等流程抽象为角色定义（Compiler Agent = 研究员，Synthesizer Agent = 分析师，Linter Agent = QA），用方法论驱动的框架思路重新设计多 Agent 协作流程。
