## 研究问题

Agent 技能（Skills）与工作流（Workflow）的交叉地带正在发生什么？当技能不再只是静态工具封装，而开始承担工作流编排、知识沉淀和自我进化的职责时，**技能与工作流的融合呈现怎样的架构分层、设计模式和演进路径？**

本综合分析基于 13 个交叉概念和 20+ 个相关概念与摘要，试图回答：从 MCP 协议的连通性到 Skill 蒸馏的知识提炼，再到达尔文.skill 的自进化系统，技能如何逐步成为工作流的核心原子？

## 综合分析

### 一、能力获取的三层光谱：从连通到封装到蒸馏

技能与工作流的融合并非一步到位，而是沿着一条清晰的抽象阶梯展开：

| **层级** | **代表概念** | **核心解决的问题** | **工作流角色** | **典型实现** |

| --- | --- | --- | --- | --- |

| L0 连通层 | MCP 协议 | AI 能连接什么工具 | 管道基础设施 | Apify MCP Server、Gate MCP Skills |

| L1 封装层 | Agent Skills、[SKILL.md](http://skill.md/) | AI 知道怎么用工具 | 可复用执行单元 | Claude 官方 Skills 仓库、Chops 统一管理 |

| L2 蒸馏层 | Skill 蒸馏、六阶段蒸馏 SOP | 从实践中提炼可复用知识 | 知识编码与沉淀 | Claude Code 源码蒸馏、仓颉.skill 整书蒸馏 |

| L3 进化层 | 达尔文.skill、AutoSkill | 技能自主优化与淘汰 | 自治工作流组件 | 8维度评估 + keep/revert 机制 |

**关键发现**：L0 到 L3 不是替代关系，而是叠加关系。MCP 解决「能不能连」，Skills 解决「会不会用」，蒸馏解决「从哪学」，进化解决「怎么变好」。一个成熟的 Agent 工作流需要四层同时在位。

### 二、技能作为工作流原子的两条路径

从源材料中可以识别出两条截然不同的「技能→工作流」转化路径：

| **维度** | **自底向上路径（CLI-first）** | **自顶向下路径（Protocol-first）** |

| --- | --- | --- |

| **起点** | 命令行工具 + pipe 组合 | MCP 协议 + schema 定义 |

| **代表** | MMX-CLI、lark-cli、dingtalk-workspace-cli | Gate MCP Skills、Apify MCP Server |

| **Token 成本** | 极低（CLI 天然精简） | 较高（schema bloat，单工具定义可达 4000+ tokens） |

| **工作流组合性** | 强（pipe/重定向/并行天然支持） | 中（需要编排层协调） |

| **生态开放性** | 弱（各家 CLI 互不兼容） | 强（协议统一，跨平台可用） |

| **实测成本差** | 每月 1 万次操作 $3.2 | 每月 1 万次操作 $55.2（17 倍） |

钉钉和飞书同时选择 CLI 路线绕开 MCP，验证了一个关键判断：**当技能需要高频嵌入工作流时，Token 效率比协议优雅更重要**。但 MCP 不会消亡——它退回到开放生态接入协议层，类似 SOAP 的演化路径。

### 三、「Thin Harness, Fat Skills」设计哲学的实证

跨概念分析揭示了一个反复出现的架构模式：**编排层（Harness）应该尽量轻薄，领域知识和执行逻辑应该下沉到 Skills 中**。

支撑证据链：

- **Agent Skills** 的设计理念：「比单个提示词更结构化，包含元数据、指令和附加资源」

- [**SKILL.md**](http://skill.md/) 同时承担路由信号与能力发现入口的双重职责

- **工作流自动化代理** 作为 Skill 使用时，常承担「把模糊任务转成 SOP」的中间层角色

- **Skill Book** 机制：从一次成功的对话中自动提炼工作流模板，实现「经验即技能」

这意味着工作流编排的复杂度不应由编排引擎承担，而应分散到各个 Skill 中。每个 Skill 自己知道何时触发、如何执行、怎样处理异常。

### 四、技能质量治理的系统化演进

随着 Skill 数量膨胀，质量治理从无到有经历了三个阶段：

1. **人工筛选期**：依赖创作者自行判断 Skill 质量（早期 [SKILL.md](http://skill.md/) 阶段）

1. **标准化评估期**：8维度评估体系（结构 60 分 + 效果 40 分）提供了可比较的量化标准

1. **自动进化期**：达尔文.skill 通过自动变异、评估和 keep/revert 实现 Skill 的持续优化

**能力图谱** 的出现尤其值得注意——它用多维度描述（任务适用性、成本、延迟、成功率）替代单一星级评分，这是 Skill 从「工具」升级为「工作流组件」的标志：工作流编排器需要比星级更丰富的信息来做路由决策。

### 五、蒸馏作为技能-工作流之间的桥梁

Skill 蒸馏是连接「静态技能封装」与「动态工作流执行」的关键桥梁。源材料中出现了三种不同粒度的蒸馏模式：

| **蒸馏模式** | **输入** | **输出** | **工作流含义** |

| --- | --- | --- | --- |

| **源码蒸馏** | 开源项目源码（如 Claude Code 51 万行） | 设计原则 → Agent Skill | 把他人最佳实践编码为自己的工作流组件 |

| **人格蒸馏** | 公开语料（文章、访谈、语录） | 心智模型 → .skill 文件 | 将专家认知转化为决策辅助工作流 |

| **整书蒸馏** | 完整书籍 | 六阶段 SOP → Skill 集合 + 关系网络 | 系统化知识提取的标准化工作流 |

六阶段蒸馏 SOP 尤其值得关注：它本身就是一个精密的工作流（整书理解 → 五 Agent 并行提取 → 三重验证 → Skill 构造 → 关系链接 → 压力测试），而它的产出又是新的工作流组件。**蒸馏工作流生产工作流组件**——这是一种递归的能力扩张模式。

## 关键发现

> **💡** **发现 1：Token 经济学是技能-工作流融合的隐形架构师**

CLI 与 MCP 的 17 倍成本差不仅是技术选型问题，它直接决定了工作流的可行粒度。高频工作流（如社群消息监控、每日情报简报）在 MCP 成本下不经济，只有 CLI 路线才能支撑。**技能的工作流化程度受限于其调用成本**，而非能力上限。

> **💡** **发现 2：蒸馏的递归性创造了「技能工厂」模式**

女娲.skill 蒸馏人格生成 .skill，仓颉.skill 蒸馏书籍生成 Skill 集合，达尔文.skill 对 Skill 自动变异和评估——这三者形成了一条完整的「技能生产技能」链路。**工作流的终极形态不是执行任务，而是生产新的执行能力**。这在传统自动化中没有对应物。

> **💡** **发现 3：Skill 评估正在从质量门禁演变为路由信号**

8维度评估体系最初是为 keep/revert 决策设计的（质量门禁），但能力图谱的出现将评估信息转化为路由依据——工作流编排器可以根据任务特征、成本约束和成功率预期选择最合适的 Skill。评估系统从「事后检查」变成「事前路由」，这是技能深度嵌入工作流的标志。

> **💡** **发现 4：「提示词工作流化」揭示了 AI 产品化的最小可行路径**

从一次性 prompt → [SKILL.md](http://skill.md/) 封装 → Skill Book 自动提炼 → 工作流自动化代理，存在一条完整的「经验资产化」路径。这条路径的每一步都在降低复用门槛：从手动编写变为半自动提炼，再到全自动蒸馏。**对个人用户而言，Skill 蒸馏就是知识管理的执行层**。

> **💡** **发现 5：浏览器控制层正在成为技能-工作流融合的新战场**

DOM 代理自动化、浏览器控制层、Chrome Skills 等概念的密集出现表明，浏览器正从「人看网页」的工具变成「Agent 操作 Web 应用」的执行环境。Google Chrome 直接引入 Skills 概念，意味着浏览器本身正在成为技能的宿主平台和工作流的执行运行时。

## 行动建议

> **🎯** **建议 1：在 OpenClaw 工作流中建立 CLI-first 技能层**

基于 CLI 比 MCP 低 17 倍成本的实证数据，Tizer 应优先将高频调用的 OpenClaw 技能封装为 CLI 接口而非 MCP Server。具体做法：参考 MMX-CLI 的三大设计原则（输出隔离、语义化状态码、非阻塞异步），为 OpenClaw 的核心内容管线技能设计 CLI wrapper。MCP 保留用于低频、需要跨平台发现的场景。

> **🎯** **建议 2：用六阶段蒸馏 SOP 系统化 Tizer 的 Skill 库建设**

当前 Skill 蒸馏多停留在单篇源码或单个工具层面。建议引入仓颉.skill 的六阶段蒸馏 SOP（整书理解 → 五 Agent 并行提取 → 三重验证 → Skill 构造 → 关系链接 → 压力测试），对 Tizer 常用的参考书籍和方法论做系统蒸馏。结合 8 维度评估体系建立 Skill 质量基线，确保每个 Skill 的「实测输出质量」分数达标后才进入正式工作流。

> **🎯** **建议 3：构建内容管线的「能力图谱」路由层**

随着 Tizer 的 Skill 库增长，单靠手动选择 Skill 将变得低效。建议为内容管线中的每个 Skill 建立能力图谱（适用任务类型、平均 Token 成本、成功率、延迟范围），并实现一个简单的路由逻辑：根据当前任务特征自动匹配最优 Skill 组合。这将把 Skill 管理从「人选工具」升级为「工作流自动路由」，是 HITL 工作流向更高自动化程度演进的关键一步。
