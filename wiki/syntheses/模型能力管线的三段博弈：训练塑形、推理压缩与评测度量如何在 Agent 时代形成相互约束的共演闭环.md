## 研究问题

训练/微调、推理优化与模型评测三个环节在传统认知中是线性流水线——先训练、再优化部署、最后评测验收。但在 Agent 时代，这三者正在形成**非线性共演闭环**：训练方式决定了推理可优化的上限，推理优化反过来约束了训练目标的有效性，而评测体系的度量偏差又在暗中塑造训练与优化的方向。**这条「能力管线」的三段之间究竟存在哪些隐性耦合、结构性矛盾和共演动力学？**

## 综合分析

### 一、训练→推理：架构决策如何预设推理优化的天花板

训练阶段的架构选择并非只影响训练效率——它直接决定了推理阶段能做哪些优化、不能做哪些优化。

| **训练架构决策** | **推理优化空间** | **推理约束** | **典型案例** |

| --- | --- | --- | --- |

| MoE 稀疏激活 | 激活参数少→本地部署可行，长上下文解码速度 7-19x | Expert parallelism 需专用调度，路由机制与推理引擎紧耦合 | Qwen3.5 397B/17B、Qwen3.6-35B-A3B |

| Dense 全参数 | 部署简单，数据并行即可 | 高推理成本，本地部署困难 | 传统 LLM |

| Attention Residuals（选择性回忆） | 深层信息不被稀释，推理质量更高 | 非标准残差连接，推理引擎需适配 | Kimi 团队架构创新 |

| Sparse Reference Attention | 参考长度计算复杂度降至近似线性 | 非对称注意力模式需定制化推理内核 | Avatar V 视频生成 |

**关键发现**：MoE 架构是训练—推理耦合的典型案例。MoE 的训练目标（大参数量、条件计算）本质上是在训练时「预编程」推理时的效率——总参数量大但激活参数少的设计，让 Qwen3.5 在消费级硬件上可运行，同时保持前沿能力。但 Expert parallelism 把模型结构设计与推理引擎调度紧密耦合，意味着训练阶段的专家分片策略直接约束了推理部署的并行方案。

### 二、推理→训练：蒸馏与 RL 如何把推理经验注入训练循环

推理优化不仅是训练的「下游消费者」，还通过蒸馏和强化学习反向影响训练过程本身。

- **蒸馏作为推理→训练的反馈通道**：Qwen3.5-27B-Claude-4.6-Opus-Reasoning-Distilled-v2 展示了一种典型模式——从闭源强模型的推理轨迹中提取知识，注入更小的开源模型。这不是简单的模型压缩，而是把推理时观察到的「思考方式」编译回训练数据。蒸馏的合规边界问题揭示了一个结构性矛盾：推理优化的最强路径（蒸馏闭源模型）与训练合规的边界直接冲突。

- **On-Policy RL 作为推理行为的直接优化器**：On-Policy RL 优化的不是静态样本上的损失函数，而是模型「实际会怎么搜、怎么答」。这意味着推理时的真实行为（包括工具调用序列、检索策略、引用决策）被直接编码进训练目标。训练与推理的边界在这里彻底模糊——推理产生的轨迹就是训练数据，训练优化的目标就是推理效果。

- **CFG+DMD 两阶段蒸馏**：在视频生成等高计算成本场景，后训练阶段的蒸馏策略（如 Avatar V 的 10x 加速）决定了产品竞争力不仅取决于生成质量，还取决于蒸馏后的成本与时延。这揭示了一个深层张力：**训练追求的「质量上限」与推理优化追求的「效率下限」之间的博弈，最终由蒸馏策略来调解**。

### 三、评测→训练/推理：度量体系如何暗中塑造优化方向

评测看似是中立的「裁判」，实际上它的度量方式深刻影响了训练和推理优化的方向选择。

| **评测范式** | **对训练的引导效应** | **对推理优化的引导效应** |

| --- | --- | --- |

| SWE-bench / Terminal-Bench 等静态基准 | 训练数据向基准倾斜，过拟合风险 | 优化为基准跑分而非真实场景 |

| LLM-as-judge 动态评审 | 训练目标转向「通过模型评审」，而非人类偏好 | 推理需兼顾评审模型的偏好模式 |

| 单变量实验循环 | 结构化搜索替代随机调参，训练效率提升 | 推理参数调优可归因、可累计 |

| TimesFM 零样本评估 | 预训练底座直接作为 baseline，降低训练投入门槛 | 推理时无需微调即可验证可行性 |

**关键发现**：LLM-as-judge 正在成为连接评测与训练的「隐性桥梁」。当评审模型被用于 RL 训练的奖励评分时（如 RULER 框架），评测体系不再是训练的外部观察者，而变成了训练循环的内部组件。这创造了一个递归问题：**用模型来评价模型训练出的模型**，评测标准本身成为训练偏差的来源。

### 四、三段闭环的结构性矛盾

三段之间存在至少三个不可调和的结构性张力：

1. **效率—能力悖论**：MoE 等稀疏架构在训练时引入条件计算以换取推理效率，但这种「预编程效率」限制了推理时的灵活性（如 Expert parallelism 的刚性分片）。训练阶段为推理效率做的每一个让步，都是对模型能力上限的一次微妙压缩。

1. **蒸馏—评测循环陷阱**：蒸馏把大模型推理能力压缩到小模型→小模型在基准上表现接近→评测体系无法区分「真正理解」和「轨迹模仿」→训练社区误判能力边界→更激进的蒸馏→更大的「Goodhart 效应」风险。

1. **评测时效性缺口**：评测基准的更新速度远慢于训练技术的迭代速度。当 On-Policy RL 已经在优化模型的实时搜索行为时，SWE-bench 等静态基准仍在测量代码补全能力——评测体系度量的「能力」与训练优化的「能力」已经出现定义分裂。

## 关键发现

1. **MoE 是训练—推理耦合的「活化石」**：MoE 不是一种推理优化技术，而是在训练阶段就预设了推理效率的架构决策。Qwen3.5 的 397B/17B 设计意味着训练时就已经为推理时的计算预算做了承诺——这种「编译时优化」模式正在取代传统的「运行时优化」思路。

1. **蒸馏正在消解训练与推理的边界**：当蒸馏从闭源模型的推理轨迹提取知识时，「训练数据」的定义已经扩展到包括推理时产生的思维链。On-Policy RL 进一步模糊这条边界——推理行为即训练信号，训练目标即推理效果。这意味着未来的「训练」可能不再是一个独立阶段，而是嵌入推理循环中的持续过程。

1. **LLM-as-judge 将评测从外部观察者变成训练内部组件**：当 LLM-as-judge 被用于 RL 奖励信号时，评测体系不再是独立的第三方验证，而成为训练优化目标的一部分。这创造了一个认识论困境：我们用什么来评价评价本身的可靠性？

1. **「基准跑分」与「真实能力」的剪刀差正在扩大**：静态基准（SWE-bench、Terminal-Bench）度量的是可控环境下的表现，但 Agent 时代的核心能力是多步工具使用、实时搜索决策和开放域问题解决。评测体系的滞后性意味着训练优化方向可能正在系统性地偏离真实需求。

1. **三段闭环催生「全栈模型工程」新范式**：训练架构→推理部署→评测反馈的紧耦合意味着单独优化任何一段都可能是次优的。MoE + On-Policy RL + LLM-as-judge 的组合预示着一种新的工程范式：在训练阶段就联合优化推理效率和评测表现，而非串行处理。

## 行动建议

1. **在 OpenClaw 的 Agent 训练流程中引入「推理效率约束」**：当前训练与推理优化是分离的两个阶段。建议在 RL 训练目标中直接加入推理成本约束（如 Token 预算、延迟上限），使训练阶段就能产出「推理友好」的行为策略，而非训练完再压缩。

1. **建立「评测—训练」隔离墙**：避免将 LLM-as-judge 的评分直接作为 RL 奖励信号的唯一来源。建议采用混合评估架构——硬指标（构建耗时、测试覆盖率）+ LLM-as-judge + 人工抽样验证的三层评审，防止评测偏差通过训练循环放大。

1. **关注 MoE 生态作为本地部署的关键路径**：Qwen3.6-35B-A3B 级别的 MoE 模型已经在消费级 GPU 上可运行。对于 Tizer 的知识管线和 Agent 系统，本地 MoE 推理可以显著降低 API 成本并提升隐私性，建议评估将部分高频低延迟任务迁移到本地 MoE 推理。
