## 定义

上下文边界分解（Context-based Decomposition）是一种多 Agent 系统架构设计原则：按**信息依赖关系**而非**角色标签**来拆分 Agent 职责。核心问题不是"该分几个 Agent"，而是"这个任务实际需要哪些上下文？"

> Design around context boundaries, not roles.

## 关键要点

- 传统做法按角色拆分（planner → developer → tester），每次 handoff 都会丢失上下文，质量逐级下降

- 正确做法：如果两个任务共享深层上下文，应该放在同一个 Agent 里；只有当上下文可以被干净地隔离时才拆分

- Sub-Agent 模式适用于上下文可隔离的独立任务——每个 sub-agent 在隔离环境中执行，只返回最终结果（压缩信号）

- Agent Teams 模式适用于上下文需要共享的强依赖任务——团队成员保持持久上下文并实时通信

- 选择架构的判断标准：任务之间是否存在信息依赖？依赖程度如何？

## 与角色分解的对比

| 维度 | 角色分解（常见但有缺陷） | 上下文边界分解（推荐） |

| --- | --- | --- |

| 拆分依据 | planner / developer / tester | 信息依赖图 |

| 上下文处理 | 每次 handoff 丢失 | 按边界精确隔离或共享 |

| 质量趋势 | 逐级衰减 | 保持一致 |
