## 📖 概述

这是知识 Wiki 系统的「宪法文件」，定义了 Wiki 的组织规则、页面模板、流程规范。

所有 Wiki 系统 Agent 和 Notion AI 都应遵循此文件。各 Agent 的详细职责分工见 [系统工作流程图](index/系统工作流程图.md)。

---

## 📂 条目类型定义

### concept（核心概念词条）

页面结构：

- **定义**：一句话说明白这个概念是什么

- **关键要点**：3-5 个核心要点

- **实例/应用**：具体使用场景或案例

- **关联概念**：与其他 concept 页面的 `<mention-page>` 链接

- **来源引用**：引用的 summary 条目列表

### summary（单篇资料摘要）

页面结构：

- **一句话摘要**：文章核心观点

- **关键洞察**：3-5 个最重要的发现/观点

- **提取的概念**：使用 `<mention-page>` 标签引用对应的 concept 页面（不是纯文本）

- **原始文章信息**：作者、来源、发布时间、原文链接

- **个人评注**：与 Tizer 现有工作流的关联点

### 🔗 引用链格式强约束（跨所有条目类型）

所有结构化引用**必须**使用 `<mention-page>` 标签，**严禁**纯文本引用。具体：

- summary 的「提取的概念」段 → `<mention-page url="...">concept 名称</mention-page>`

- concept/entity 的「来源引用」段 → `<mention-page url="<summary-url>">摘要：<原文标题></mention-page>`

- synthesis 的「来源列表」→ `<mention-page>` 引用所有 summary/concept

- qa 的「引用来源」 → `<mention-page>` 引用引用的 Wiki 页面

原因：结构化引用是知识图谱连通、孤岛检测、状态晋升的基础。Compiler 编译时必须遵守；历史纯文本存量由 **Wiki 引用升级员** 自动清理。

### synthesis（跨资料综合分析）

**何时创建**：

- **单标签**：某个标签下积累了 **≥10 个 concept/entity**，且没有已有 synthesis 覆盖该主题

- **双标签**：两个标签的交集 **≥3 个共享 concept/entity**（如「Agent 编排 × 记忆系统」）

- **三标签（高阶综合 / 面）**：三个标签各自拥有 **≥5 个 concept/entity**，且三个标签的两两交集各至少有 **≥2 个共享 concept/entity**。三标签 synthesis 应引用已有的双标签 synthesis 作为输入，在其基础上提炼跨三边的涌现结构（闭环、矛盾、收敛模式等双标签视角无法呈现的洞察）

- **四标签（立体综合 / 体）**：在一个已有的三标签三角形（A、B、C）基础上，引入第四个「远距离」标签 D，要求：

  - 已存在以 A、B、C 为主题的三标签 synthesis（或至少 2 篇双标签 synthesis 覆盖了该三角形）

  - D 与 A、B、C 的交集应很小（各交集 **≤3 个**共享 concept/entity，即「远距离」）

  - D 自身拥有 **≥5 个 concept/entity**

  - 写作要求：Synthesizer 不是找已有连接，而是强制在三角形与 D 之间寻找**结构同构、隐喻迁移、或范式冲突**。若无法找到至少 1 个结构同构，则不产出（质量门槛）

- 或者一轮系统性对话产出了重要的架构决策

**产出配额**：每次最多生成 **1 篇单/双标签** + **1 篇三标签** + **1 篇四标签**，深度优先。优先级：四标签（体）＞ 三标签（面）＞ 双标签（线）＞ 单标签（点）。

**扫描逻辑**：Synthesizer 每次运行时，先检查是否存在可产出的四标签组合（以已有的三标签 synthesis 为锚点，遍历剩余标签找远距离候选），再筛选三标签组合，最后扫描双标签和单标签候选。

页面结构：

- **研究问题**：这篇 synthesis 要回答什么问题

- **综合分析**：跨多篇资料的综合推理（鼓励使用对比表）

- **关键发现**：3-5 个跨来源才能发现的洞察

- **来源列表**：引用的所有 summary 和 concept

- **行动建议**：基于分析的可执行建议

### qa（问答沉淀）

**执行者**：Wiki QA Agent

**何时创建**：当回答综合了 **≥3 个 Wiki 页面**的内容，且问题不是一次性的操作请求（如「帮我合并这两个概念」）。

页面结构：

- **问题**：原始提问

- **回答**：综合回答正文

- **引用来源**：使用 `<mention-page>` 引用的 Wiki 页面列表

### entity（实体档案 — 人物/产品/公司的持续跟踪页）

**与 concept 的区别**：

- concept = 技术概念、方法论、架构模式（如 Harness 工程、ReAct Agent、RAG）

- entity = 具体的产品、公司、人物（如 OpenClaw、Anthropic、Karpathy）

**判断标准**：如果这个概念有版本历史、融资信息、GitHub Stars、团队背景等「档案属性」，它就是 entity 而不是 concept。

页面结构：

- **简介**：一句话介绍这个实体

- **关键数据**：官网、GitHub、Stars、License、团队、融资等

- **版本/动态时间线**：重要版本发布和事件记录（持续更新）

- **与我的关系**：Tizer 如何使用/关注这个实体

- **相关 concept**：与此实体相关的技术概念 `<mention-page>` 链接

- **所有引用的 summary**：所有提到这个实体的摘要列表

### lint-report（健康报告）

页面结构：

- 检查日期、健康度评分、孤岛条目、过期草稿、重复疑似、状态异常、标签异常、标签分布统计、状态晋升建议、改进建议

## 📨 对话产出沉淀规则

当 Notion AI 与用户的对话产出了有价值的结果时，按以下规则判断是否需要沉淀到 Wiki：

| 对话类型 | 是否入 Wiki | 沉淀方式 |

| --- | --- | --- |

| **知识检索**（“X 是什么？”） | ✅ 是（综合≥3页时） | 由 Wiki QA Agent 回答并自动创建 `qa` 页面 |

| **系统设计/架构决策** | ✅ 是 | 提炼决策备忘录为 `synthesis`，规则本身已物化到 Agent 指令 |

| **日常维护执行**（合并重复、晋升状态） | ❌ 否 | Lint Report 已记录，无需额外页面 |

| **头脑风暴/探讨** | 视情况 | 有结论的提炼为 concept 或 synthesis，纯发散不入库 |

## 🔄 Ingest 流程（资料编译）

**执行者**：Wiki Compiler Agent

**触发条件**：新文章入库（page.created，60s 防抖）或手动 @mention。~~定时批量触发已关闭（存量已清零）~~

**执行步骤**：

1. 读取新文章的完整内容和备注

1. **🔒 Summary 去重检查**：先用源文章 URL 查询；若未命中且存在外部原文链接，再用「标题候选 + 原文链接核验」做第二层去重；对 `Learn more in the official documentation` 这类导航/占位标题或正文主要为目录跳转的页面直接跳过，不创建 summary

1. 创建一条 **summary** 页面（状态=已审核）

1. 从文章中提取核心概念（3-7 个）

1. 对每个概念：

  - **强制查重**：先将名称归一化（转小写、去空格、去标点），用 SQL 精确匹配 + 模糊匹配查询已有 concept

  - 已有 → 只追加引用，不创建新页面

  - 没有 → 创建新 concept 页面（状态=草稿）

1. 提取的概念用 `<mention-page>` 回填到 summary 的「提取的概念」段

1. 设置所有新页面的标签、置信度、最后编译时间

1. 标记源文章 `已编译到Wiki = true`

**重要原则**：

- 增量更新，不要全量重写已有页面

- concept 名称使用业界通用术语，遵循命名归一化规则

## 🔍 Query 流程（检索回答）

**主要执行者**：Wiki QA Agent（知识库专属问答窗口）、Notion AI（兑底 + 系统管理场景）

**其他 Agent 也使用 SQL 查询来完成各自职责**（Compiler 查重、Lint 统计、Fixer 定位目标等）。

**查询策略（按优先级）**：

1. **SQL 直查**：用 `querySql` 按名称、标签、类型等条件查询数据库，这是最快最准的方式

1. **读取具体页面**：用 `loadPage` 读取 concept/summary/synthesis 的完整内容

1. **全局搜索**：如果 SQL 查询结果不足，用 `search` 跨所有来源搜索

**回答流程**：

1. 根据问题类型选择查询策略，定位相关页面

1. 读取页面完整内容

1. 综合推理，标注来源引用。**根据来源条目状态差异化处理**：

  - **已审核**：优先引用，作为核心论据

  - **审核中**：正常引用

  - **草稿**：标注「此概念为新建条目，内容可能不完整」

  - **需更新**：标注「此内容可能已过时，建议交叉验证」

1. 如果回答具有沉淀价值，按「对话产出沉淀规则」判断是否创建 qa 或 synthesis

**常用 SQL 模板**：

- 按标签找 concept：`SELECT url, "名称" FROM wiki WHERE "类型" = 'concept' AND "标签" LIKE '%推理优化%'`

- 按名称模糊查找：`SELECT url, "名称", "类型" FROM wiki WHERE "名称" LIKE '%MemBrain%'`

- 查看最近更新：`SELECT url, "名称", "类型" FROM wiki ORDER BY "最后编辑时间" DESC LIMIT 10`

- 查看已有 synthesis：`SELECT url, "名称" FROM wiki WHERE "类型" = 'synthesis'`

## 🩺 Lint 流程（健康检查）

**执行者**：Wiki Lint Agent

**触发条件**：recurrence 每日 14:00 自动执行，或手动 @mention

### Phase 1：快速检查（纯 SQL，全量覆盖）

不需要 loadPage，全部通过 `querySql` 完成，保证 100% 条目覆盖：

1. **全库统计** — 按类型×状态交叉统计

1. **过期草稿检测** — 草稿超过 **7 天**：内容完整 → 建议晋升为审核中；内容不完整 → 报警

1. **时效性检查** — 最后编译时间超过 **30 天**或为 null（豁免 qa 类型和系统元概念）

1. **完全同名重复检测** — `GROUP BY "名称", "类型" HAVING COUNT(*) > 1`

1. **近似重复检测（归一化）** — 分两步：

  - Step A：SQL 查出所有 summary/concept/entity 的名称和 url

  - Step B：在推理层做归一化匹配（lowercase → 去标点/空格/句号 → 统一繁简体 → 去版本号后缀），输出疑似重复对

1. **状态异常** — 状态为空的页面

1. **标签异常** — 标签为空或使用已废弃标签 + 全量标签分布统计

1. **类型启发式筛查** — SQL 全量查出 concept 名称，在推理层用规则批量筛查「疑似应为 entity」：

  - 规则 A：名称含版本号（`2.0`/`v3`）或产品后缀（`Tool`、`SDK`、`CLI`等）

  - 规则 B：首字母大写英文专有名词、大写缩写+产品名组合

  - 输出疑似列表，标注「建议人工确认」

### Phase 2：深度检查（需 loadPage，抽样覆盖）

从 Phase 1 发现的问题条目 + 随机抽样中执行：

1. **引用结构化检查** — 分层抽样 ≥80 个草稿 concept/entity（每创建日 ≥8 条），检查来源引用是否使用 `<mention-page>` 结构化链接。若 affected page rate ≥ 30% 或 plain-text rate ≥ 20%，报告标注「🚨 系统性问题」并推荐 Wiki 引用升级员批处理（而非 Fixer 逐条）

1. **标签分类合理性抽查** — 加载 Phase 1 筛查标记的条目 + 随机抽样，人工判断标签匹配度

1. **状态晋升评估** — 综合 Phase 1 数据，判断哪些条目应晋升（草稿→审核中：7天+内容完整；草稿/审核中→已审核：≥2篇引用；审核中/已审核→需更新：>30天）

### 输出规范

lint-report 必须包含：

- 健康评分（0-100）+ 计分明细表格

- 各检查项的发现（含具体页面链接）

- **Fixer 可执行的结构化修复清单**（每条标注：修复类型 + 目标页面 URL + 具体操作）

- 待办建议（需人工介入的项目）

- 报告末尾 @mention Wiki Fixer 触发自动修复

### Fixer 协作协议

- 建议分两类：🤖 **自动修复项**（重复合并、标签/类型修正、引用结构化升级）+ 👤 **人工介入项**（Schema 变更、豁免规则、Compiler 调整）

- Fixer 完成后应在 Lint Report 追加「修复记录」区块，列出每项修复的操作和结果

- Recheck 时，Lint Agent 只需读取修复记录 + 验证变更页面属性，无需全量重扫

**SQL 规范**：不允许自连接、多条语句、动态 LIKE 拼接。必须分步查询（先查 concept/entity 列表，再查 summary 列表），在推理层做匹配。

## 📝 命名规范

- **concept**：英文产品名跟官方大小写一致；中文用最短准确名；不加括号说明；版本号不单独建页；别名选一个标准名做标题

- **summary**：「摘要：」+ 原文标题

- **synthesis**：研究问题作为标题

- **entity**：实体官方名称（产品用官方写法如 OpenClaw、LangGraph；人物用常用英文名如 Karpathy；公司用官方名如 Anthropic）

- **lint-report**：「Lint Report YYYY-MM-DD HH:mm」（必须含时分，避免同一天多份报告同名）

## 🏷️ 三维度标签体系

### 维度框架

标签分为三个维度，每个条目优先从不同维度组合分配标签：

**A. 场景领域**（用在哪里？解决什么现实问题？）

量化交易 | 稳定币 | 加密资产 | 链上协议 | 知识管理 | 社交媒体 | AI 设计 | 身份准入 | 内容自动化 | AI 政策 | 商业/生态 | 前端开发 | Agent 协作模式

**B. 技术方法**（核心技术手段是什么？）

推理优化 | 训练/微调 | 模型评测 | 多模态 | 模型部署 | 代码生成 | 图像生成 | 视频/3D | RAG/检索 | 上下文管理 | 长期记忆 | MCP 协议 | 多Agent协作 | 浏览器自动化 | 提示工程 | AI 对齐 | Agent 安全 | Harness 工程

**C. 产品形态**（长什么样？怎么交付？）

CLI 工具 | IDE 集成 | 笔记工具 | AI 产品 | 算力基础设施 | OpenClaw

### 标签规则

1. 优先分配 2-3 个不同维度的标签；纯技术机制/概念若只适合 1 个维度，1-2 个标签也可以，不要硬凑

1. **如果某个维度确实不适用，留空比硬凑更好。错误标签的危害远大于缺少标签**

1. 标签可自适应生长——现有标签池无法描述的内容可创建新标签，须明确归属 A/B/C 维度、中文优先、粒度对齐现有标签

1. 每个标签只在确实相关时使用，宁缺毋滥

### 已退休标签（禁止使用）

以下 11 个旧大标签因覆盖面过宽、信号密度低，已退休并由上述细粒度标签替代：

~~LLM~~ | ~~Agent 框架~~ | ~~Agent 编排~~ | ~~Agent 技能~~ | ~~Coding Agent~~ | ~~开发工具~~ | ~~工作流~~ | ~~记忆系统~~ | ~~安全/隐私~~ | ~~Crypto/DeFi~~ | ~~内容创作~~

另有早期废弃标签：~~AI Agent~~ | ~~MCP~~ | ~~Notion~~

### 易混淆标签定义

- **知识管理**(A)：专指知识管理方法论、PKM 工具链、笔记体系设计等「如何组织和管理知识」的场景。**不是万能兜底标签**——如果一个概念只是「可能对知识工作有用」但核心不是在讲知识管理本身，就不要打这个标签。例：Obsidian 插件生态 ✅、Zettelkasten 方法论 ✅、稀疏注意力 ❌、One-Eval ❌、Metric Library ❌。判断标准：核心论述是否围绕「知识的组织、存储、检索、流转」展开？

- **笔记工具**(C)：面向终端用户的笔记/知识管理产品（如 Obsidian、Notion、Logseq），不包括 Agent 内部的文件存储或 memory 机制

- **多Agent协作**(B)：多 Agent 之间的技术实现层——消息传递、任务分发、并行执行等

- **Agent 协作模式**(A)：多 Agent 的组织设计与场景——角色分工、群聊交互、竞争/协作动力学等

- **Harness 工程**(B)：围绕模型构建系统层能力的工程方法论——执行环境、验证机制、文件系统、安全边界等整体脚手架。不包括组织行为或 Agent 动力学

- **模型部署**(B)：大模型的部署与 serving（如 Ollama、vLLM、TensorRT），不包括嵌入式编程

### 按条目类型的标签参考

- **entity**（实体）→ 侧重 A + C

- **concept**（概念）→ 侧重 A + B，纯技术概念只有 B 也 OK

- **summary**（摘要）→ 侧重 A + B/C

- **synthesis**（综合）→ 尽量 3 个维度都覆盖

### 二级标签（来源标签，细粒度）

存储在「来源标签」属性中，是从源文章数据库（微信文章/X书签）透传的原始标签，不做映射和归一化。

- **用途**：保留原始信息供查阅和追溯，支持细粒度 SQL 搜索

- **格式**：字符串，如「OpenClaw, Agent, LLM, 自动化, Claude, 记忆」

- **维护者**：Compiler 在编译时自动从源文章的标签属性复制

## 🤖 Agent 职责速查

| Agent | 角色 | 核心职责 | 可修改范围 |

| --- | --- | --- | --- |

| **Wiki Compiler** | 编译器 | 文章 → summary + concept | 创建新 summary/concept，追加引用 |

| **Wiki Lint Agent** | 诊断师 | 每日 14:00 健康检查 → lint-report | 只创建报告，不修改任何页面 |

| **Wiki Fixer** | 治疗师 | 按 Lint Report 自动修复重复合并、类型迁移、标签修复、小量引用升级等决策密集型修复 | 状态/标签/类型修复、同名重复合并、近似重复合并；不得修改 index 类型系统页；删除需人类指示。大规模纯文本引用升级交给 Wiki 引用升级员，状态晋升交给 Wiki 状态晋升员 |

## 🔄 变更同步规则

当任何 Agent 的指令页面被修改时，必须同步检查并更新以下文件：

**谁负责**：执行修改的角色（Notion AI 或用户）在修改 Agent 指令后，应立即执行同步。不允许「只改指令不同步宗法文件」。
